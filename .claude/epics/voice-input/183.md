# Task 012: ChatProvider — audio plumbing and voice-error surfacing

## Context

`ChatProvider` is the state machine that owns the chat lifecycle: building requests, subscribing to the SSE stream, parsing the `message--//--[records_json]` response, and saving records to the database. Voice input adds a new input path (audio bytes) and a new output condition (empty records after an audio send = interpretation failure). This task threads audio through the provider and implements the detection logic for the "I didn't catch that" error message (AD-5 from the epic).

## Acceptance Criteria

- [ ] **FR-4 / Voice-only:** Calling `sendMessage('', audioBytes: nonEmptyBytes)` results in a request with `audio` in the body and empty-string `query` (verified via mock ChatApiService).
- [ ] **FR-4 / No audio:** Calling `sendMessage('hello')` behaves identically to before this task (no regression).
- [ ] **FR-4 / Empty bytes guard:** Calling `sendMessage('', audioBytes: Uint8List(0))` is treated as a no-op (same as `sendMessage('')` with no image — the existing AD-6 guard).
- [ ] **FR-5 / Voice error shown:** When audio was sent AND the parsed records array is `[]`, the assistant message's `content` is replaced with the localized `voice_didnt_catch_that` string (not the raw AI text).
- [ ] **FR-5 / Voice error not triggered without audio:** When no audio was sent AND records are `[]` (a normal non-financial chat reply), the assistant message content is kept as-is (the raw AI text).
- [ ] **FR-5 / Voice with records:** When audio was sent AND records are non-empty, the normal record-card flow runs; no error string is shown.
- [ ] **No regression:** Existing `sendMessage(content, imageBytes: ...)` call sites still compile and behave identically.

## Technical Details

- **`hadAudio` as local variable (AD-5):** Store as a local in `_handleStream` rather than an instance field. The streaming guard (`_isStreaming`) prevents concurrent sends, but local variables are cleaner and don't require reset logic.
- **`ImageProcessingService().toBase64` reuse:** This method does `base64Encode(bytes)` — it works on any byte array. No audio-specific encoding is needed client-side; the AAC file produced by `AudioRecordingService.stop()` is already in the correct format.
- **Mutually exclusive audio+images:** The UI (task 010) never sets both `audioBytes` and `imageBytes` simultaneously (PRD constraint). The provider does not enforce this — it's enforced by the UI. Both fields can coexist in the method signature for future flexibility.
- **Empty-records for non-audio chat:** A normal text reply that the AI returns with `[]` records (e.g., "Hello! How can I help?") must NOT show the voice error. The `hadAudio` guard is the sole discriminator.
- **Translate fallback:** `_localeProvider?.translate(key) ?? 'hardcoded'` is the existing pattern. If `translate` returns null or the key is missing, the hardcoded English string is shown.
