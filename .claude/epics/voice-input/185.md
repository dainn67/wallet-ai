# Task 090: Integration tests, cross-device smoke, and docs

## Context

This is the final quality gate for the voice-input epic. All prior tasks (001–013) must be closed before this one starts. This task verifies that all components integrate correctly end-to-end — the chat tab triggers recording, the service captures audio, the provider packages it, the API service ships it, and the correct UI state is shown in both success and failure scenarios. It also covers NFR-2 (recording latency) via manual device testing, and updates the living-docs that CLAUDE.md mandates be kept current.

## Acceptance Criteria

- [ ] **FR-1 / Regression:** All pre-existing chat tests continue to pass; no regressions from voice-input changes.
- [ ] **FR-2 / Integration — happy path:** Integration test drives mic-tap → (simulated) stop → fake ChatApiService returns records → record card visible in chat.
- [ ] **FR-2 / Integration — 30-s auto-stop:** Fake `AudioRecordingService` fires `onAutoStopped`; integration test confirms message is sent automatically.
- [ ] **FR-3 / Integration — cancel:** Fake service cancel → no message sent → `ChatProvider.sendMessage` not called.
- [ ] **FR-5 / Integration — voice error:** Fake stream returns `'Sorry--//--[]'` with `hadAudio == true` → assistant message shows `voice_didnt_catch_that` string.
- [ ] **FR-5 / Integration — image error:** Force image processing exception in test → snackbar with `image_load_failed` string appears.
- [ ] **NFR-2 / Latency — manual:** On a real device, tap mic → recording bar visible ≤ 300 ms (confirmed via visual stopwatch or DevTools timeline).
- [ ] **Full build green:** `fvm flutter build apk --debug` and `fvm flutter build ios --no-codesign` succeed.
- [ ] **Docs:** `docs/features/voice-input.md` created; `project_context/context.md` updated with voice-input flow; `project_context/architecture.md` updated to mention `AudioRecordingService`.

## Implementation Steps

### Step 1: Create integration test suite

- Create `tests/integration/epic_voice_input/voice_input_test.dart`.
- Follow the pattern of existing integration tests in `tests/integration/` (if any exist — check for examples).
- Use `testWidgets` with `pumpWidget` to build the full `ChatTab` with mocked providers.
- Mock both `AudioRecordingService` and `ChatApiService` via their `setMockInstance` hooks.

Test scenarios to implement (each as a separate `testWidgets` block):

**Scenario A — Happy path (voice send → record card):**
1. Pump `ChatTab` with mock `ChatProvider` + mock `AudioRecordingService`.
2. Tap mic icon → verify `AudioRecordingService.start()` called.
3. Tap stop (■) → verify `AudioRecordingService.stop()` called; returns `Uint8List.fromList([1,2,3])`.
4. Mock `ChatApiService.streamChat` returns a fake SSE: `'Got it--//--[{"source_id":1,"amount":"50000","category_id":1,"description":"lunch","type":"expense","suggested_category":null,"occurred_at":null}]'`.
5. Pump until settling → verify record card is visible in the chat list.

**Scenario B — Cancel:**
1. Pump, tap mic → start.
2. Tap × (cancel) → verify `AudioRecordingService.cancel()` called.
3. Verify `ChatProvider.sendMessage` NOT called.
4. Verify chat list unchanged.

**Scenario C — Auto-stop:**
1. Pump, tap mic.
2. Fire `onAutoStopped` callback directly from the mock service.
3. Verify `ChatProvider.sendMessage` called with non-null `audioBytes`.
4. Verify recording bar disappears.

**Scenario D — Voice error (empty records):**
1. Pump, tap mic → stop.
2. Mock `streamChat` returns `"Sorry I didn't understand--//--[]"`.
3. Pump until settling → verify assistant message content == `"I didn't catch that. Please try again."`.
4. Verify NO record card shown.

**Scenario E — Streaming guard:**
1. Pump with `isStreaming == true`.
2. Verify mic icon's `onTap` is null.
3. Simulate tap → nothing happens.

**Scenario F — Image error (retroactive):**
1. Pump, mock `ImageProcessingService.compress` to throw.
2. Trigger image pick flow.
3. Verify `SnackBar` with `"Couldn't load the image. Please try again."` appears.

### Step 4: Update docs/features/voice-input.md

- Create `docs/features/voice-input.md`.
- Document:
  - **Feature overview:** What it does, who it's for.
  - **User flow:** Mic tap → recording bar → stop → AI processes audio → record card (or error).
  - **Technical flow:** `ChatTab._onMicTap` → `AudioRecordingService.start()` → `_stopAndSend()` → `ChatProvider.sendMessage(audioBytes:)` → `ImageProcessingService.toBase64` → `ChatApiService.streamChat(audioBase64:)` → `/streaming` endpoint → Gemini audio inline part → `message--//--[records_json]` → parse → `RecordProvider.createRecord`.
  - **Error paths:** Voice failure (empty records + hadAudio), image failure (processing exception → snackbar).
  - **Constraints:** 30-second max, audio-only (no combined voice+image), server routing deferred.
  - **Package:** `record ^5.x`, AAC-LC 128 kbps mono.
