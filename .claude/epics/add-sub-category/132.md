# Task: Provider Hierarchy Logic

## Context
Update `RecordProvider` to correctly handle the hierarchical structure of categories for easier UI consumption.

## Interface Contract
- **Receives from T130**: Hierarchical category data in the database.
- **Produces for T133**: A structured map of parent ID to sub-categories.

## Acceptance Criteria
- [ ] `RecordProvider` populates a `Map<int, List<Category>> subCategories`.
- [ ] `getCategoryName` correctly returns "Parent - Sub" if a parent exists.
- [ ] Provider notifies listeners when categories change.

## Implementation Steps
1.  **Modify `lib/providers/record_provider.dart`**:
    *   Add `Map<int, List<Category>> _subCategories = {};`.
    *   Add getter: `List<Category> getSubCategories(int parentId) => _subCategories[parentId] ?? [];`.
    *   Update `loadAll`:
        *   After fetching categories, filter those with `parentId != -1`.
        *   Populate `_subCategories` by grouping sub-categories by their `parentId`.
    *   Update `getCategoryName`:
        *   Find category by ID.
        *   If `parentId != -1`, fetch the parent name and return `${parentName} - ${c.name}`.
        *   Otherwise, return just `${c.name}`.

- Run `fvm flutter test test/providers/record_provider_test.dart` (ensure hierarchical display logic is tested).
- Check that the `subCategories` map is correctly populated in debug logs.
