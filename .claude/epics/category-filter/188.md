# Task: RecordProvider data layer — sort fix + category filter method

## Context

Two small changes to `lib/providers/record_provider.dart` that unblock all downstream tasks. First: the Records tab currently sorts by `recordId DESC` (insertion order), causing edited records to jump position. The fix is a one-line sort key swap to `occurredAt DESC` — the field already exists on the model (added in schema v8). Second: the new category drill-down popup needs a pure in-memory filter that returns records for a given set of category ids within the tab's current month — no new DB query, just filtering `_records` which is already fully loaded.

## Description

Modify `RecordProvider.filteredRecords` to sort by `occurredAt DESC` instead of `recordId DESC`. Add a new `getRecordsForCategory(List<int> categoryIds, DateTimeRange? range)` method that filters `_records` by category membership and date range, returning results sorted `occurredAt DESC`. Both changes are additive with no impact on existing provider consumers other than the sort order improvement.

## Acceptance Criteria

- [ ] **FR-4 / Stable position after edit:** Given two records A (occurredAt=today) and B (occurredAt=last month), when record B is edited without changing its `occurredAt`, B remains below A in `filteredRecords`.
- [ ] **FR-4 / New record ordering:** A new record created with today's `occurredAt` appears first in `filteredRecords`.
- [ ] **FR-1 / Single category filter:** `getRecordsForCategory([2], range)` returns only records with `categoryId == 2` within the range, sorted `occurredAt DESC`.
- [ ] **FR-2 / Union filter:** `getRecordsForCategory([1, 2, 3], range)` returns all records whose `categoryId` is in {1, 2, 3}, sorted `occurredAt DESC`.
- [ ] **NFR-1 / Null safety:** Records where `categoryId` is null or -1 are never incorrectly included when not in the `categoryIds` list.

## Implementation Steps

### Step 1: Fix sort key in `filteredRecords`
- Open `lib/providers/record_provider.dart`
- Find the sort block (around line 112): `filtered.sort((a, b) { return b.recordId.compareTo(a.recordId); });`
- Replace with:
  ```dart
  filtered.sort((a, b) => b.occurredAt.compareTo(a.occurredAt));
  ```
- `occurredAt` is `int` (millisecondsSinceEpoch) on the `Record` model — `compareTo` on ints gives the correct descending order when reversed.

### Step 2: Add `getRecordsForCategory` method
- In `lib/providers/record_provider.dart`, after the `getCategoryTotal` method (around line 63), add:
  ```dart
  List<Record> getRecordsForCategory(List<int> categoryIds, DateTimeRange? range) {
    final r = range ?? _selectedDateRange;
    return _records.where((rec) {
      if (!categoryIds.contains(rec.categoryId)) return false;
      if (r == null) return true;
      final occurred = DateTime.fromMillisecondsSinceEpoch(rec.occurredAt);
      return !occurred.isBefore(r.start) && !occurred.isAfter(r.end);
    }).toList()
      ..sort((a, b) => b.occurredAt.compareTo(a.occurredAt));
  }
  ```
- This is a pure read — no `notifyListeners()` call.
- Falls back to `_selectedDateRange` when `range` is null (popup uses the tab's current month automatically).

### Step 3: Verify no existing consumers break
- Run `fvm flutter analyze lib/providers/record_provider.dart` — no new errors.
- Check that `filteredRecords` is still called the same way in `lib/screens/home/tabs/records_tab.dart` and any other consumers — the signature is unchanged, only the sort order differs.

## Technical Details

- **`Record.occurredAt`** is `int` (millisecondsSinceEpoch), present since schema v8, backfilled from `lastUpdated` for existing rows. Safe to sort on without null check.
- **`Record.categoryId`** is `int?` — `contains` on `List<int>` correctly returns false for null.
- **Pattern to follow:** The existing `filteredRecords` getter shows the filter + sort pattern; `getRecordsForCategory` follows the same structure.
- **No DB impact:** `_records` is loaded at startup via `loadAll()`; this method does not touch SQLite.
