## Implementation Steps

### Step 1: Run literal-hardcoding grep
```bash
grep -rEn \
  'Color\(0x[0-9A-Fa-f]{8}\)|Colors\.[a-zA-Z]+[^t]|fontSize:\s*[0-9]|borderRadius:\s*BorderRadius\.circular\([0-9]' \
  lib/components/ lib/screens/ \
  --include="*.dart"
```
Expected: zero matches (or only `Colors.transparent` and `Colors.black` in shadow — those are acceptable).

**If violations found:**
- Identify the responsible task file by the file path in the match.
- Fix inline in the `lib/` file — do not create a new branch.
- Re-run grep until clean.

### Step 2: Run full test suite
```bash
fvm flutter test
```
Expected: zero failures.

If failures on `suggestion_banner_test.dart`: check that lines 130 and 156 were updated in T5.
If failures elsewhere: trace to the task that touched those files, fix, re-run.

### Step 3: Verify test diff is scoped correctly
```bash
git diff main..HEAD -- test/
```
Expected output: changes only in `test/components/suggestion_banner_test.dart`, exactly 2 lines changed (the `FilledButton` → `TextButton` lines). Any other test file changes are a violation — investigate and revert.

### Step 4: google_fonts audit
```bash
grep -rn "google_fonts" lib/ pubspec.yaml
```
Expected: zero matches.

### Step 5: Orphan file audit (final check)
```bash
find lib/components lib/screens -name "*.dart" | while read f; do
  name=$(basename "$f" .dart)
  if ! grep -rl "$name" lib/ --include="*.dart" | grep -v "^$f\$" | grep -q .; then
    echo "ORPHAN: $f"
  fi
done
```
Expected: zero orphans. If any found, delete and re-run T9's cleanup.

### Step 6: Cold-start performance trace
1. Save a baseline if not already done:
   ```bash
   flutter run --profile --trace-startup 2>&1 | grep "Engine appeared"
   ```
   Run on a real device or fixed-spec emulator. Record the startup time in milliseconds.
2. Compare against the pre-epic baseline (captured before T1 started, or use `git stash` to revert and measure, then re-apply).
3. Difference must be ≤ +100ms.
4. If ≥ +100ms delta: profile which build step or widget tree path increased startup. The most likely cause is the `ThemeExtension` registration — it runs in `AppTheme.light()` at app start. If it's the bottleneck, consider `const` constructors for the extension.

## Technical Details
- **Approach:** No new code. Grep + test run + device testing.
- **`Colors.transparent` exclusion in grep:** The regex `Colors\.[a-zA-Z]+[^t]` excludes `Colors.transparent` (ends in `t`) but catches `Colors.white`, `Colors.black`, `Colors.purple`, etc. Double-check: `Colors.black.withValues(alpha: 0.04)` in shadow is acceptable (excluded from NFR-1 per epic spec).
- **Performance baseline:** If no baseline was captured before T1, use `git stash` to temporarily revert to `main` state, measure, then `git stash pop`. Or compare with the git tag `pre-redesign-ui` if it was created.
- **If NFR-4 fails:** The most likely offenders are: (1) `AppSemanticColors` with 6-color `categoryAccents` list initialized at startup — make it `const`; (2) any widget tree pre-warm triggered by the new `ThemeData`. Profile with `flutter devtools` startup timeline.
