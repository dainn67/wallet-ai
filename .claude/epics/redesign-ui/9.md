# T9: Onboarding content + TestTab refactor + orphan cleanup

## Context
After all tab and popup migrations (T5-T8) are complete, three clean-up concerns remain: (1) `OnboardingDialog` step content should use the new component styles — T8 handled the dialog's visual chrome; this task updates the step content widgets inside each step. (2) `TestTab` is a developer tool — it needs its existing buttons/cards/tiles swapped to themed equivalents so it doesn't use old pre-redesign widgets. (3) After 10 migration tasks, some `.dart` files in `lib/components/` or `lib/screens/` may have become orphaned (not imported by anything). This task audits and deletes them. This is also the moment to confirm no commented-out pre-redesign code blocks remain.

## Description
Three distinct sub-tasks:
1. **Onboarding step content** — update each step in `OnboardingDialog` to use `IconSquare`, `SectionLabel`, `AppColors`, themed buttons for step CTAs.
2. **TestTab refactor** — swap any stock Material buttons/cards in `test_tab.dart` to their themed counterparts.
3. **Orphan file audit** — run import-graph scan, delete any unreferenced `.dart` files, remove commented-out old widget code.

## Acceptance Criteria
- [ ] **FR-9 / Onboarding CTAs use themed buttons:** All "Next", "Skip", "Finish" CTAs in `OnboardingDialog` use `FilledButton` (primary) or `TextButton` (secondary) from the theme — no inline `ButtonStyle`.
- [ ] **FR-9 / Onboarding step content:** Step headers use `Theme.of(context).textTheme.headlineSmall` + `AppColors.primary` color; step illustrations or icon placeholders use `IconSquare` where applicable.
- [ ] **FR-10 / TestTab uses themed widgets:** `TestTab` has no `ElevatedButton`, `RaisedButton`, `FlatButton`, or widgets with hardcoded `color:` arguments for their own fill.
- [ ] **NFR-6 / Zero orphaned files:** `grep` import-graph audit finds no `.dart` files in `lib/components/` or `lib/screens/` that are not imported by at least one other file.
- [ ] **NFR-6 / No dead comment blocks:** No commented-out multi-line widget code remains in `lib/components/` or `lib/screens/` from pre-redesign.
- [ ] **NFR-2 / No regressions:** `fvm flutter test` passes after all three sub-tasks.

## Implementation Steps

### Step 1: Onboarding step content (`onboarding_dialog.dart`)
T8 migrated the dialog's outer chrome (container, navigation buttons). This step updates step content:
1. Read the current steps array or step-builder in `onboarding_dialog.dart`.
2. For each step:
   - Header text: wrap in `Text(title, style: Theme.of(context).textTheme.headlineSmall?.copyWith(color: AppColors.primary, fontWeight: FontWeight.w600))`.
   - Body text: `Theme.of(context).textTheme.bodyMedium`.
   - Step icon/illustration: if a `CircleAvatar` or `Icon` is used as a step illustration, replace with `IconSquare(icon: stepIcon, tint: AppColors.primary, size: 64)`.
   - CTA buttons: confirm T8 already replaced them with `FilledButton`/`TextButton` — if not, do it here.
3. Do NOT change: step count, step order, the logic that checks `isFirstLaunch`, the `onComplete` callback.

### Step 2: TestTab refactor (`test_tab.dart`)
1. Read `lib/screens/home/tabs/test_tab.dart`.
2. Find all `ElevatedButton`, `RaisedButton`, `FlatButton`, or `TextButton(style: ...)` with hardcoded colors. Replace with:
   - Test action buttons → `OutlinedButton` (secondary feel) or `FilledButton` (primary) depending on the action's intent.
   - Any hardcoded `color: Colors.xyz` on a container → `color: AppColors.primary` or `Theme.of(context).colorScheme.primaryContainer`.
3. Typography: replace any `TextStyle(fontSize: 14, ...)` hardcoded styles with `Theme.of(context).textTheme.bodyMedium`.
4. Do NOT change: the test actions themselves (what each button does), the test state display logic, or any `kDebugMode` guard.

### Step 3: Orphan file audit
Run the import graph scan:
```bash
# Find all .dart files in lib/components/ and lib/screens/
find lib/components lib/screens -name "*.dart" | while read f; do
  base=$(basename "$f")
  # Check if any other .dart file imports this file directly or via barrel
  if ! grep -rl "$base\|$(basename "$f" .dart)" lib/ --include="*.dart" | grep -v "^$f\$" | grep -q .; then
    echo "ORPHAN: $f"
  fi
done
```
For each `ORPHAN:` file reported:
1. Manually confirm it is truly unused (not referenced via string interpolation or reflection — unlikely in this codebase but check).
2. Verify it is not a barrel file that exports other files.
3. If confirmed orphan: delete the file.
4. Remove its export from any barrel (e.g., `lib/components/components.dart`).

**Also scan for commented-out blocks:**
```bash
grep -rn "// TODO: old\|// OLD:\|// REMOVE:\|// pre-redesign\|\/\*.*old.*\*\/" lib/components/ lib/screens/ --include="*.dart"
```
Delete any commented-out multi-line pre-redesign widget code found.

- **Approach:** Cleanup and polish pass. Most changes are small (text style, button type).
- **Pattern to follow:** Follow the same theming approach used in T5–T8. The onboarding steps are effectively mini popups — same design rules apply.
- **Orphan scan note:** The barrel-file convention means most files are exported from `components.dart` or similar. A file is a true orphan only if the barrel *also* has no importers — check the barrel import chain.
- **`grep -v "^$f\$"` in the orphan scan:** Excludes self-reference. On macOS, `grep -rl` output path format may differ — test the command manually before batch-deleting.
- **Safety on deletion:** Before deleting any file, run `grep -r "import.*$(basename $f .dart)" lib/` as a double-check.
