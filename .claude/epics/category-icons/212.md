# Task: Server SuggestedCategory.emoji + validation (wallyai)

## Context
When the WalletAI chat AI proposes a new category in the suggestion banner, it currently returns `name`, `type`, `parent_id`, and `message`. To enable per-category emoji on the client, the server must return an additional `emoji` glyph in the JSON. AI output is unreliable — the model may return `"transport"`, `""`, an emoji-of-the-week, multi-grapheme strings, or omit the field entirely — so the server is the authoritative validator. Without this, the client can never receive a real AI-picked emoji and would always use the fallback `🏷️`.

## Description
In the sibling repo `../chatbot-flow-server/` (wallyai project scope only — per CLAUDE.md, ignore AIKaze, LinguaAI, Math Bear), extend the `SuggestedCategory` JSON output schema with an `emoji` field. Update the category-detection prompt to instruct the model to return a single emoji glyph that best represents the proposed category, with one in-prompt example. Add a server-side validator step (between the LLM output and the JSON returned to the client): trim whitespace, accept only if the trimmed value contains 1 grapheme cluster (≤8 codepoints) AND contains at least one codepoint in the Unicode emoji ranges (e.g., U+1F300–U+1FAFF, U+2600–U+27BF, supplementary symbols/pictographs). On any failure → set `emoji = '🏷️'` before returning. Add WalletAI-only tests for valid + invalid LLM outputs.

## Acceptance Criteria

- [ ] **FR-2 / Happy path:** Given a chat message that triggers a new-category suggestion, the server returns a JSON `SuggestedCategory` whose `emoji` field is exactly one emoji glyph (e.g., `"emoji": "🚌"` for a transport-like category).
- [ ] **FR-2 / Existing fields preserved:** The existing `name`, `type`, `parent_id`, `message` fields on `SuggestedCategory` are unchanged in shape and contract — no test in the wallyai suite asserting current behavior regresses.
- [ ] **NFR-1 / AI returns non-emoji text:** Given the LLM returns `"emoji": "transport"`, the server rewrites the field to `"emoji": "🏷️"` before sending.
- [ ] **NFR-1 / AI returns empty / missing:** Given the LLM returns `"emoji": ""`, `"emoji": null`, or omits the field entirely, the server's response contains `"emoji": "🏷️"`.
- [ ] **NFR-1 / AI returns multi-emoji or sentence:** Given the LLM returns `"emoji": "🚌🚕"` or `"emoji": "🚌 bus"`, the server rewrites to `"emoji": "🏷️"` (grapheme count != 1 OR has non-emoji codepoints).
- [ ] **NFR-1 / AI returns valid emoji with skin-tone / ZWJ:** Given the LLM returns a ZWJ sequence like `"emoji": "👨‍💻"` (≤8 codepoints, single grapheme), the server accepts and returns it unchanged.

## Technical Details

- **Approach:** Mirrors epic AD-4 (server validates and coerces; both sides defensive).
- **Why server-side validation is non-negotiable:** the client must be able to assume `suggestion.emoji` is always a renderable single-grapheme emoji string. If the client also tried to validate, the AI's bad output would still leak through the wire format and could be cached/logged downstream.
- **Why ZWJ allowed up to 8 codepoints:** modern compound emoji (family, profession, skin-tone modifiers) routinely use 5–7 codepoints joined with U+200D. 8 is a safe ceiling per the PRD.
- **Edge cases:**
  - LLM returns `null` vs missing field vs empty string — all map to `'🏷️'` via the trim-then-empty-check.
  - LLM returns a flag emoji (e.g., `🇻🇳`) — regional indicators are 2 codepoints, valid single grapheme, accept.
  - LLM hallucinates Unicode in the wrong range (e.g., a single CJK character) — rejected by the emoji-range check.
- **Patterns to follow:** the existing wallyai validators (in whatever layer they live — look for similar `name`/`type` normalization in the suggestion pipeline).
