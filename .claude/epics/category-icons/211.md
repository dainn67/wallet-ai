# Task: Add emoji column, migration & curated seed map

## Context
Every category in WalletAI currently renders as plain text only, making the Records and Categories tabs feel flat next to the redesigned chat surfaces. This task adds the data foundation вЂ” a single `emoji` field on `Category` plus a SQLite v9в†’v10 migration that backfills the curated emoji for every seed category вЂ” so that no `Category` instance can ever exist without an emoji. Without this, every downstream task (server contract, suggestion-banner persist, edit dialog, render sites) has nothing to read or write.

## Description
Add `String emoji` (non-null, default `'рџЏ·пёЏ'`) to the `Category` model вЂ” round-trip via `toMap`/`fromMap`/`copyWith`. Bump `_dbVersion` from `9` to `10` in `RecordRepository`. Update the `CREATE TABLE Category` DDL in `_onCreate` to include `emoji TEXT NOT NULL DEFAULT 'рџЏ·пёЏ'`. Update `_seedDatabase` so every seed `db.insert` carries the curated emoji per the epic's AD-5 map. Add `RecordMigrationService.addEmojiColumn(db)` that mirrors the idempotent pattern of `addOccurredAtColumn` (PRAGMA-guarded column check, ALTER TABLE ADD COLUMN with constant default, then guarded UPDATE per seed `category_id` that only writes where `emoji = 'рџЏ·пёЏ'` so user values are never overwritten). Wire it from `_onUpgrade` via `if (oldVersion < 10)`. Sub-categories (Groceries, Dining Out, Taxi, Fuel, Cinema, Streaming, Clothes, Electronics) are matched by `name` + `parent_id` since their ids are not pinned.

## Acceptance Criteria

- [ ] **FR-1 / Happy path (new install):** Fresh-install opens DB at version 10; `SELECT name, emoji FROM Category WHERE parent_id = -1` returns every parent seed with the AD-5 emoji (Uncategorizedв†’рџЏ·пёЏ, Foodв†’рџЌ”, Transportв†’рџљ—, Entertainmentв†’рџЋ¬, Salaryв†’рџ’°, Rentв†’рџЏ , Healthв†’рџЏҐ, Shoppingв†’рџ›ЌпёЏ, Transferв†’рџ”„).
- [ ] **FR-1 / Sub-category seed:** `SELECT name, emoji FROM Category WHERE parent_id != -1` returns the 8 sub-category emojis (Groceriesв†’рџ›’, Dining Outв†’рџЌЅпёЏ, Taxiв†’рџљ•, Fuelв†’в›Ѕ, Cinemaв†’рџЋҐ, Streamingв†’рџ“є, Clothesв†’рџ‘•, Electronicsв†’рџ“±).
- [ ] **FR-1 / Default for user-created category:** Inserting a `Category` without specifying emoji (or with `null`) results in `emoji = 'рџЏ·пёЏ'` on disk.
- [ ] **FR-1 / Model round-trip:** `Category.fromMap(c.toMap())` returns a Category equal to the original (including emoji); `copyWith(emoji: 'рџљЊ')` returns a new Category with the new emoji and all other fields preserved; `fromMap` of a map missing `emoji` returns `Category(emoji: 'рџЏ·пёЏ')`.
- [ ] **FR-6 / v9в†’v10 backfill:** Starting from a v9 DB with the original seed names and no emoji column, running the migration adds the column and backfills the AD-5 emoji per seed (matched by `category_id` for parents, by `(name, parent_id)` for subs).
- [ ] **FR-6 / Idempotent migration:** Running `addEmojiColumn` twice on the same DB is a no-op the second time: column exists (PRAGMA check skips ALTER), and the UPDATE's `WHERE emoji = 'рџЏ·пёЏ'` guard prevents overwriting any non-default value.
- [ ] **FR-6 / User value preserved:** If a row in a v9 DB somehow has a non-default emoji (e.g., dev hack), the v10 migration does NOT overwrite it.

## Technical Details

- **Approach:** Mirrors AD-1 (single column, not a join table), AD-2 (ADD COLUMN + guarded UPDATE, no rebuild), AD-5 (concrete seed map).
- **Existing pattern to follow:** `RecordMigrationService.addOccurredAtColumn` (lib/services/record_migration_service.dart:19) вЂ” same idempotency shape (PRAGMA check, ALTER if missing, post-ALTER UPDATE with guard).
- **Files to create/modify:**
  - `lib/models/category.dart` вЂ” extend with `emoji` field + round-trip
  - `lib/repositories/record_repository.dart` вЂ” bump version, update DDL, update seed inserts, hook migration into `_onUpgrade`
  - `lib/services/record_migration_service.dart` вЂ” add `addEmojiColumn` helper + seed-map constants
  - 3 test files as above
- **Edge cases:**
  - User-created Category from before v10 has `emoji = 'рџЏ·пёЏ'` (the column default) вЂ” that's correct behavior per PRD US-3.
  - A pre-v10 row whose `parent_id = -1` and `name` happens to match a sub-category name (e.g., a custom "Taxi" parent) must NOT be touched by the sub backfill вЂ” the `WHERE parent_id != -1` clause guarantees this.
  - The `Transfer` category may already exist from v8в†’v9's defensive insert вЂ” the parent backfill matches by `category_id = 9`, but the v8в†’v9 migration may have assigned a different id for upgraders. If id 9 is missing in their DB, the parent emoji UPDATE is a no-op for that row, and that's acceptable вЂ” they don't have a stable seed id to anchor to.
- **Why the `WHERE emoji = 'рџЏ·пёЏ'` guard matters:** the column's `DEFAULT 'рџЏ·пёЏ'` populates every existing row at ALTER time, so the post-ALTER UPDATE always runs on default-only rows. The guard is defensive insurance for future migrations or test setups that pre-populate emojis.
