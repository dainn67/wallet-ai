# Task: Parse server emoji on client and persist on confirm

## Context
The server (task 212) now returns an `emoji` field on `SuggestedCategory`, and the `Category` model (task 211) carries an `emoji` column. The remaining link is the client's parse + persist path: the chat stream's suggestion JSON must be parsed into `SuggestedCategory.emoji`, and when the user taps "Confirm" on the in-bubble `SuggestionBanner`, the new `Category` row must be created with that emoji. Without this task, AI-suggested emojis are decoded by the server, sent over the wire, and dropped on the client floor.

## Description
Extend `SuggestedCategory` with a non-null `String emoji` field and a `fromJson` that reads `json['emoji']` with a `'🏷️'` fallback for null/missing/empty (defensive belt-and-braces with the server validation per AD-4). In `ChatProvider`, find the helper that resolves a `SuggestedCategory` to either an existing `Category` (by name + type + parent_id) or creates a new one — pass `suggestion.emoji` into the `Category` constructor on the create branch. The resolve-existing branch must NOT overwrite an existing category's emoji (matches the migration's WHERE-guard philosophy: never silently overwrite a user value). Add unit tests covering the parse fallback paths and the confirm-flow persist.

## Acceptance Criteria

- [ ] **FR-3 / Happy path:** Given a suggestion JSON `{"name":"Commute","type":"expense","parent_id":-1,"message":"...","emoji":"🚌"}`, `SuggestedCategory.fromJson` returns an instance with `emoji == "🚌"`.
- [ ] **FR-3 / Confirm persists emoji:** Given the user taps Confirm on a banner whose suggestion has `emoji = '🚌'` and the name does not match any existing category, a new `Category` row is created with `emoji = '🚌'`.
- [ ] **FR-3 / Existing category not overwritten:** Given a suggestion whose name+type+parent_id matches an existing `Category` with `emoji = '🍔'`, and the suggestion's emoji is `'🍕'`, the existing category's emoji remains `'🍔'`.
- [ ] **NFR-1 / Defensive fallback (missing field):** `SuggestedCategory.fromJson({...no emoji...})` returns `emoji == '🏷️'` (does not throw, does not return null).
- [ ] **NFR-1 / Defensive fallback (empty / null):** `fromJson({...,"emoji": ""})` and `fromJson({...,"emoji": null})` both return `emoji == '🏷️'`.
- [ ] **NFR-1 / Existing contract unchanged:** Existing `fromJson` invariants hold — `name` empty → returns `null`; `type` not in `{'expense','income'}` → returns `null`; valid input returns a `SuggestedCategory` (now with `emoji`).

## Implementation Steps

### Step 1: Extend `SuggestedCategory`
- Modify `lib/models/suggested_category.dart`:

  - Update constructor: add `required this.emoji` — but since the factory `fromJson` is the only construction site for parsed data and it always coalesces to `'🏷️'`, prefer `this.emoji = '🏷️'` default so tests/manual code can construct without passing it.
  - Update `fromJson`:
    ```dart
    final rawEmoji = json['emoji'] as String?;
    final emoji = (rawEmoji != null && rawEmoji.isNotEmpty) ? rawEmoji : '🏷️';
    ```

    Pass `emoji: emoji` into the `SuggestedCategory(...)` constructor call.
  - Update `toString()` to include the emoji.

## Technical Details

- **Approach:** Mirrors AD-4 — client treats `emoji` as best-effort, defensive fallback to `'🏷️'`. Mirrors AD-1 — `emoji` lives on `Category`, suggestion just carries it through.
- **Patterns to follow:** existing `SuggestedCategory.fromJson` (lib/models/suggested_category.dart:14) — same null-safe parse pattern (`json['x'] as String?`) with a sensible default; same `try/catch` envelope.
- **Edge cases:**
  - JSON parser yields `int` instead of `String` for `emoji` (e.g., LLM hallucinates a number) — `as String?` cast fails inside the `try` block → existing `catch (_)` returns `null`. That's consistent with how malformed `name` is handled today: the whole suggestion is rejected rather than partially salvaged. Acceptable.
  - "Existing category not overwritten" is a deliberate design choice — alternative (always update) was rejected because it would silently erase a user's manual emoji choice every time the AI re-suggested the same name.
