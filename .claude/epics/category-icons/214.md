# Task: Emoji input in category form dialogs

## Context
Users need a way to set or change the emoji on a category — to correct an AI-picked choice they disagree with, or to assign one to a category that migrated with the fallback `🏷️`. The PRD specifies a plain `TextField` + OS emoji keyboard as the primary UX (per epic AD-3), explicitly deferring `emoji_picker_flutter`. Both dialogs (parent and sub) need the same input. The Uncategorized category (id=1) remains non-editable, consistent with existing behavior.

## Description
Add a single "Emoji" `TextField` next to the existing "Name" field in both `lib/components/popups/category_form_dialog.dart` (parent categories) and `lib/components/popups/add_sub_category_dialog.dart` (sub-categories). The field shows the current emoji as its initial value, with a live emoji preview rendered adjacent so users see the result before saving (this preview also serves as the hint that the field expects an emoji). On save: validate non-empty AND contains at least one emoji codepoint; if invalid or empty → coerce to `'🏷️'`. Persist via the existing repository update path using `Category.copyWith(emoji: ...)` from task 211. Disable the input for `categoryId == 1` (Uncategorized), matching the existing "not editable" pattern. No new pub.dev dependency — relies on the OS-native emoji keyboard.

## Acceptance Criteria

- [ ] **FR-4 / Happy path (parent dialog):** Given a parent category with `emoji = '🍔'`, opening the edit dialog shows `🍔` as the initial value in the Emoji field. Replacing it with `🍕` and tapping Save persists the category with `emoji = '🍕'`.
- [ ] **FR-4 / Happy path (sub dialog):** Same as above for a sub-category in `add_sub_category_dialog.dart`.
- [ ] **FR-4 / Live preview:** Typing/selecting a new emoji in the field updates the adjacent preview before save.
- [ ] **FR-4 / Cleared field coerces to fallback:** Clearing the Emoji field and tapping Save persists `emoji = '🏷️'` (the category is never empty).
- [ ] **FR-4 / Invalid input coerced:** Entering plain text (e.g., `food`) and tapping Save persists `emoji = '🏷️'`.
- [ ] **FR-4 / Uncategorized not editable:** The edit dialog for `categoryId == 1` either does not show the Emoji field or shows it disabled — consistent with how Name is handled today.
- [ ] **NFR-1 / Never persist empty:** Under any path through Save, the persisted `Category.emoji` is non-empty.

## Implementation Steps

### Step 1: Add the Emoji TextField to `category_form_dialog.dart`
- Read the existing dialog to understand layout, save-button wiring, and TextEditingController usage.
- In the dialog's `State`:
  - Add `late final TextEditingController _emojiController;` initialized with `widget.category?.emoji ?? '🏷️'` in `initState`.
  - Dispose it in `dispose()`.
- In the build method, next to the existing Name field, add a `Row` (or column on narrow widths) containing:
  - A small `Text` widget rendering the current value of `_emojiController` at font size ~24, wrapped in `AnimatedBuilder` listening to the controller — this is the live preview.
  - A `TextField` with `controller: _emojiController`, `maxLength: 8`, `decoration: InputDecoration(labelText: 'Emoji', counterText: '', hintText: '🏷️')`. No `keyboardType` override (OS emoji key on default keyboard).
  - For `categoryId == 1`, set `enabled: false` on the TextField (and/or skip rendering the row, matching the existing Name disable pattern).
- Add a private helper `String _coerceEmoji(String raw)`:
  - `final trimmed = raw.trim();`
  - `if (trimmed.isEmpty) return '🏷️';`
  - Use the same emoji-range check as task 212 (codepoint in U+1F300..U+1FAFF ∪ U+2600..U+27BF ∪ U+2300..U+23FF ∪ U+1F000..U+1F0FF) — if at least one codepoint matches, accept; else return `'🏷️'`. Keep grapheme-count check lenient (≤8 codepoints) to allow ZWJ sequences. Implementation tip: iterate `raw.runes` and check each codepoint against a `bool _isEmojiCodepoint(int cp)` helper with hardcoded ranges. No new package.
- Wire `_coerceEmoji(_emojiController.text)` into the Save handler — pass it to `category.copyWith(emoji: ...)` before calling the repository's update method.

### Step 2: Add the same Emoji TextField to `add_sub_category_dialog.dart`
- Mirror Step 1 in `lib/components/popups/add_sub_category_dialog.dart`. Sub-categories don't have a "Uncategorized" id=1 case, so no disable branch needed (but defensively still gate on `categoryId != 1` to match the parent dialog's pattern).
- Extract `_coerceEmoji` and `_isEmojiCodepoint` into a shared place. Recommend: put them in a small private utility in the same `popups/` directory (e.g., `lib/components/popups/_emoji_input_util.dart`) and import in both dialogs. Mark with `// ignore_for_file: ...` if linter complains about the underscore prefix. (Alternative: put in `lib/helpers/emoji_helper.dart` if you prefer the public-ish helper pattern — both are acceptable. Pick the one consistent with the repo's existing helper conventions; lean toward `lib/helpers/` since that's the canonical home per project-structure.md.)

### Step 3: Wire repository update
- The dialogs already call into a repository update path on save (existing code). Confirm the update path uses `Category.copyWith(...)` from task 211 (which now accepts `emoji`). If the update path bypasses `copyWith` and writes a Map directly, ensure the `emoji` key is included in the map.

## Technical Details

- **Approach:** Per AD-3, no `emoji_picker_flutter` dependency. The OS keyboard's emoji key is the input device. The live preview compensates for the lack of a custom picker UI.
- **Patterns to follow:**
  - `lib/components/popups/edit_record_popup.dart` and `lib/components/popups/transfer_popup.dart` for TextEditingController + dispose + Save patterns.
  - The existing Name field in `category_form_dialog.dart` for label/decoration styling — match its `InputDecoration` for visual consistency.
- **Edge cases:**
  - User types a long string of emoji (more than 8 codepoints) — `maxLength: 8` prevents overflow.
  - User types ZWJ-rich emoji (e.g., 👨‍👩‍👧) — single grapheme but 5 codepoints. The 8-codepoint cap accommodates this.
  - User pastes non-emoji text — accepted by the TextField (no IME restriction), then coerced to `'🏷️'` on Save.
  - User pastes emoji + text (e.g., `🍕 pizza`) — coerce iterates codepoints; finds the emoji codepoint, accepts. **Decision: keep this lenient.** The whole-string trim doesn't enforce "must be ONLY emoji"; the repo persists the raw trimmed string. If we want stricter, the helper would return only the first grapheme cluster. Defer the stricter mode unless QA flags it.
