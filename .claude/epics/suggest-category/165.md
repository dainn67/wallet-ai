# Task: Integrate SuggestionBanner into chat_bubble + wire Confirm/Cancel

## Context

With the model layer (T001), resolver helper (T002), stream parsing (T010), and banner widget (T011) all in place, this task connects everything: modifying `chat_bubble.dart` to render a `SuggestionBanner` below each record card that has a non-null `suggestedCategory`, and providing the Confirm/Cancel callbacks that orchestrate the full create-and-reclassify sequence (FR-4) or the no-op dismiss (FR-5).

## Interface Contract

### Receives from T010:
- `Record.suggestedCategory` is non-null for records whose `category_id` was `-1` and server returned a valid suggestion.
- `Record.recordId` is positive (set after `createRecord` awaits in `_handleStream` â€” confirmed at line 221 of `chat_provider.dart`).

### Receives from T002:
- `RecordProvider.resolveCategoryByNameOrCreate(name, type, parentId)` â†’ `Future<int?>` (returns `null` on error, already shows toast).

### Receives from T011:
- `SuggestionBanner(record, messageId, suggestion, onConfirm, onCancel)` widget â€” stateful, with internal `_isProcessing` double-tap guard.

## Acceptance Criteria

- [ ] **FR-3 / Banner visible:** In the chat, each record card with `record.suggestedCategory != null` has a `SuggestionBanner` rendered directly beneath it. Records without suggestion show only `RecordWidget` â€” no regression.
- [ ] **FR-3 / Stream guard:** During active streaming (message not yet complete), no banner is shown (suggestion state only exists post-stream â€” verify by checking when `suggestedCategory` is set vs when the widget builds).
- [ ] **FR-4 / Confirm new category:** Tapping Confirm on a banner for a new category name â†’ `resolveCategoryByNameOrCreate` called â†’ record's `categoryId` updated in DB â†’ `chatProvider.updateMessageRecord` called â†’ banner disappears â†’ record shows updated category name.
- [ ] **FR-4 / Confirm existing category:** Tapping Confirm for a name that already exists â†’ no new category created â†’ record updated with existing `categoryId` â†’ banner disappears.
- [ ] **FR-4 / Confirm error:** If `resolveCategoryByNameOrCreate` returns `null` (error case, toast already shown) â†’ `updateRecord` is NOT called â†’ banner remains visible â†’ Confirm re-enabled.
- [ ] **FR-5 / Cancel:** Tapping Cancel â†’ banner disappears â†’ record in DB is unchanged (`categoryId` remains `-1`) â†’ no category created.
- [ ] **FR-5 / Cancel no DB write:** `recordProvider.updateRecord` is NOT called during Cancel.

### Step 3: Implement `_handleConfirm` callback

Add a private method (or local function) in the widget scope:

```dart
Future<void> _handleConfirm(
  BuildContext context,
  Record record,
  String messageId,
) async {
  final recordProvider = context.read<RecordProvider>();
  final chatProvider = context.read<ChatProvider>();
  final suggestion = record.suggestedCategory!;

  final newId = await recordProvider.resolveCategoryByNameOrCreate(
    suggestion.name,
    suggestion.type,
    suggestion.parentId,
  );

  if (newId == null) return; // error toast already shown by resolver; banner stays

  final updatedRecord = record.copyWith(
    categoryId: newId,
    clearSuggestedCategory: true,
  );
  await recordProvider.updateRecord(updatedRecord);
  chatProvider.updateMessageRecord(messageId, updatedRecord);
}
```

### Step 4: Implement `_handleCancel` callback

```dart
void _handleCancel(BuildContext context, Record record, String messageId) {
  final chatProvider = context.read<ChatProvider>();
  final updatedRecord = record.copyWith(clearSuggestedCategory: true);
  chatProvider.updateMessageRecord(messageId, updatedRecord);
  // No DB write â€” categoryId remains -1
}
```

### Step 5: Verify `context` is safe at callback time

- `_handleConfirm` is `async` and uses `context.read<...>()`. Ensure `context.mounted` is checked before any `context.read` call that happens AFTER an `await`:
  ```dart
  if (!context.mounted) return;
  final chatProvider = context.read<ChatProvider>();
  ```
- Apply after the `await recordProvider.updateRecord(updatedRecord)` line.

## Technical Details

- **Approach:** Minimal change to `chat_bubble.dart` â€” swap `map` for `expand`, add two local handler methods, add one import.
- **Files to modify:** `lib/components/chat_bubble.dart` only.
- **Patterns to follow:** Existing `context.read<RecordProvider>()` usage in the file (or in similar components). See `lib/components/popups/category_form_dialog.dart` for the `updateRecord` + provider pattern.
- **Edge cases:**
  - `record.recordId` may theoretically be 0 if `createRecord` failed silently; `updateMessageRecord` uses `recordId` to find the record in `ChatMessage.records` â€” if 0, the record won't be found and the banner won't disappear. This is an acceptable degradation (the DB write also failed, so the record was never saved properly).
  - Multiple records in one message with the same suggested category name: confirming the first creates the category; confirming the second hits the name-lookup in `resolveCategoryByNameOrCreate` and reuses it â€” the duplicate guard works automatically.
  - `message.records` nullability: already guarded by the outer `if (message.records != null && message.records!.isNotEmpty)` check present in the existing code (line ~76).
