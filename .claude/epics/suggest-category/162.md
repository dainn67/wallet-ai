# Task: resolveCategoryByNameOrCreate helper on RecordProvider

## Context

When the user confirms a suggested category, the app needs to (1) check whether a category with that name already exists (to avoid duplicates), and (2) if not, create it and get its new ID. The existing `RecordProvider.addCategory()` returns `Future<void>` — the caller cannot know the new `categoryId`. This task adds a new helper that either finds an existing category by name or creates a new one, returning the `categoryId` in both cases.

## Description

Add `Future<int?> resolveCategoryByNameOrCreate(String name, String type, int parentId)` to `RecordProvider`. The method searches the cached `_categories` list first (case-insensitive name + parentId match) — if found, returns the existing ID immediately. If not found, it calls `_repository.createCategory(...)` directly and returns the newly inserted row id from sqflite. This requires verifying that `RecordRepository.createCategory` returns `int` (sqflite's `db.insert` already returns the id — it may just not be propagated yet); if it currently returns `void`, change the repository method to return `int` without breaking the existing `addCategory` caller (which can simply ignore the returned value). After creating, the in-memory category cache is refreshed.

Architecture decision AD-2: new helper rather than changing `addCategory` signature, to avoid breaking existing callers.

## Acceptance Criteria

- [ ] **FR-4 / New top-level category:** Calling `resolveCategoryByNameOrCreate('Streaming', 'expense', -1)` when 'Streaming' does not exist → creates a new category and returns a positive integer `categoryId`.
- [ ] **FR-4 / Existing category reuse:** Calling `resolveCategoryByNameOrCreate('Food', 'expense', -1)` when 'Food' already exists in `_categories` → returns the existing `categoryId` WITHOUT calling `_repository.createCategory`.
- [ ] **FR-4 / Case-insensitive match:** `resolveCategoryByNameOrCreate('food', ...)` matches an existing 'Food' category (case-insensitive).
- [ ] **FR-4 / Sub-category creation:** Calling with `parentId = 4` (existing parent) → creates a sub-category under that parent and returns its new `categoryId`.
- [ ] **FR-4 / Schema drift fallback:** Calling with a `parentId` that does not exist in `_categories` → falls back to `parentId = -1` (top-level), logs a debug message, and still creates and returns a valid `categoryId`.
- [ ] **FR-4 / Duplicate guard:** Two sequential calls with the same `name` + `parentId` → second call returns the same ID as the first (reuses, does not create a duplicate category in the DB).
- [ ] **FR-4 / Cache refresh:** After a new category is created, `RecordProvider.categories` (the public getter) contains the newly created category when accessed immediately after.
- [ ] **FR-4 / Error handling:** If `_repository.createCategory` throws → method returns `null` (no crash), and an error toast is displayed (matching existing `_performOperation` error pattern).

### Step 2: Add `resolveCategoryByNameOrCreate` to `RecordProvider`

- Add the method after `addCategory` (around line 283 in `record_provider.dart`):
  ```dart
  Future<int?> resolveCategoryByNameOrCreate(
    String name,
    String type,
    int parentId,
  ) async {
    // 1. Duplicate guard: search in-memory cache first
    final existing = _categories.firstWhereOrNull(
      (c) =>
          c.name.toLowerCase().trim() == name.toLowerCase().trim() &&
          c.parentId == parentId,
    );
    if (existing != null) return existing.categoryId;

    // 2. Schema drift: validate parentId refers to an existing category
    int resolvedParentId = parentId;
    if (parentId != -1) {
      final parentExists = _categories.any((c) => c.categoryId == parentId);
      if (!parentExists) {
        debugPrint(
          '[SuggestCategory] parent_id $parentId not found locally — falling back to top-level',
        );
        resolvedParentId = -1;
      }
    }

    // 3. Create and return new id
    try {
      final newId = await _repository.createCategory(
        Category(name: name, type: type, parentId: resolvedParentId),
      );
      await _refreshCategories(); // refresh cache
      return newId;
    } catch (e) {
      _showErrorToast('Failed to create category');
      return null;
    }
  }
  ```

- For `_showErrorToast`, follow the existing toast pattern used in `_performOperation`'s `showToastOnError` path.
