## Context

The widget's camera deep-link (T003) needs to call the same camera picker as the in-app ChatTab camera button. Today that logic lives inside `ChatTab` and is not reachable from `HomeScreen`. Extracting it into `ChatProvider` makes it callable from both the widget deep-link path and the existing in-app button without behaviour change.

## Description

Move the camera-picker code from `lib/screens/home/tabs/chat_tab.dart` into a new public method on `ChatProvider`:

```dart
Future<void> pickImageFromCamera({BuildContext? context}) async {
  // 1. check/request camera permission
  // 2. open image_picker with ImageSource.camera
  // 3. dispatch resulting XFile to the existing chat-image upload path
}
```

The in-app camera button must then call this method instead of its current private handler. The `context` parameter is optional — when called from the widget path it may be `null`, in which case permission-denied dialogs are skipped (the OS permission prompt still fires once via `permission_handler`).

T001 already located the exact line range — refer to `docs/features/home-widget.md` for the file:line reference.

## Acceptance Criteria

- [ ] **FR-3 / Camera picker reachable from provider:** `ChatProvider.pickImageFromCamera()` exists, is public, and opens the system camera when called.
- [ ] **FR-4 / Headless-safe:** Method works when called with `context: null` (the widget cold-start path) — no `BuildContext`-dependent code runs; permission denial is silent (no dialog).
- [ ] **No regression:** Tapping the camera button inside ChatTab still works exactly as before (same permission flow, same upload behaviour).
- [ ] **Permission flow:** First-time permission request still shows the OS prompt; subsequent denials show the existing in-app dialog ONLY when `context != null`.

## Implementation Steps

### Step 1: Read the current handler
- Open `lib/screens/home/tabs/chat_tab.dart` at the line range recorded by T001 in `docs/features/home-widget.md`.
- Identify: permission check, `image_picker` call (with `ImageSource.camera`), and the post-pick handler (probably calls into `ChatProvider` to attach the image and send).

### Step 2: Add `pickImageFromCamera` to ChatProvider
- Modify `lib/providers/chat_provider.dart`.
- Add public method:

```dart
Future<void> pickImageFromCamera({BuildContext? context}) async {
  final granted = await Permission.camera.request().isGranted;
  if (!granted) {
    if (context != null && context.mounted) {
      // show existing denial dialog (extract from chat_tab if needed)
    }
    return;
  }
  final picker = ImagePicker();
  final XFile? file = await picker.pickImage(source: ImageSource.camera);
  if (file == null) return; // user cancelled
  await _attachImageAndSend(file); // existing internal method, or extract
}
```

- If `_attachImageAndSend` doesn't exist yet as an internal method, extract the post-pick logic from `chat_tab.dart` into a private method on `ChatProvider` so it's reusable.

### Step 3: Update ChatTab camera button to call the provider
- In `chat_tab.dart`, replace the inline camera-picker handler with `context.read<ChatProvider>().pickImageFromCamera(context: context)`.
- Remove the now-dead code (the inline permission check + image_picker call).
- Keep any UI state local to ChatTab (e.g. loading spinner) — wire it via `Consumer<ChatProvider>` if needed.

### Step 4: Update import barrel (if applicable)
- If `lib/providers/providers.dart` doesn't re-export `ChatProvider`, add it so T003 can `import 'package:wallet_ai/providers/providers.dart'` and access the new method.

## Technical Details

- **Approach:** Reference epic AD-3 (camera trigger on ChatProvider, not global event bus).
- **Files to create/modify:**
  - `lib/providers/chat_provider.dart` — add public method `pickImageFromCamera`; optionally extract internal `_attachImageAndSend` helper.
  - `lib/screens/home/tabs/chat_tab.dart` — replace inline camera handler with provider call.
  - `lib/providers/providers.dart` (conditional) — export `ChatProvider` if not already exported.
- **Patterns to follow:** Existing async methods on `ChatProvider` (e.g. message-send flow) use `notifyListeners()` after state mutation — apply the same pattern after image attach.
- **Edge cases:**
  - Camera permission permanently denied — `Permission.camera.request()` returns `denied` without showing UI. Caller-side dialog only fires when `context != null`. Widget path is silent.
  - `image_picker` returns `null` (user cancelled mid-picker) — return early, no state change.
  - Two rapid taps on the camera button — current behaviour is "open one picker, ignore the second tap." Preserve this by guarding with a `_pickingImage` boolean if not already present.
