## Context

T003 added the deep-link branch for `homeWidget://camera`. T004 added `ChatProvider.pickImageFromCamera()`. This task is the integration: verify the **cold-start** path (app not running when widget is tapped) actually reaches the camera picker, not just the **warm-resume** path (app already in background). Cold-start is the higher-risk case because Flutter engine init, provider construction, and the widget intent all happen in the same first frame.

## Description

Two things to verify and harden:

1. **Cold-start URI capture:** `HomeWidget.initiallyLaunchedFromHomeWidget()` returns the launching Uri. It's already called in `HomeScreen.initState`, but we must confirm it correctly surfaces `homeWidget://camera` and not just `homeWidget://record`.
2. **Provider readiness:** When the widget tap triggers a cold start, `ChatProvider` may not be fully constructed when `_handleWidgetClick` fires. Wrap the dispatch in `addPostFrameCallback` (the same pattern used by onboarding and notification permission) so the call lands after the first frame and after providers are ready.

If the provider is still not ready after the first frame (race), add a single 200 ms `Future.delayed` retry. Document this fallback as a defensive measure — if it ever fires, there's a deeper issue worth investigating.

## Acceptance Criteria

- [ ] **FR-4 / Cold start to camera:** App is force-stopped. User taps the camera icon on the 2×2 widget. Result: app launches → onboarding skipped (if already completed) → ChatTab → camera picker opens. No additional taps required.
- [ ] **FR-4 / Cold start to text input:** Same as above with the write icon / bar. Result: ChatTab + input focused.
- [ ] **FR-4 / Warm resume to camera:** App is backgrounded (not stopped). User taps camera icon. Result: app comes to foreground → ChatTab → camera picker opens.
- [ ] **No regression:** App launched from launcher icon (no widget) lands on default screen as before.

## Implementation Steps

### Step 1: Audit the current cold-start path
- Open `lib/screens/home/home_screen.dart`
- Find the existing `HomeWidget.initiallyLaunchedFromHomeWidget().then((Uri? uri) { ... })` call in `initState`.
- Verify it routes through `_handleWidgetClick`. If not, route it.

### Step 2: Wrap dispatch in addPostFrameCallback
- Modify `_handleWidgetClick` (or its caller) so the actual dispatch runs inside `WidgetsBinding.instance.addPostFrameCallback((_) { ... })`.
- Add a `mounted` check inside the callback.
- This ensures the first frame is built (providers constructed, tab controller initialized) before we try to navigate or call a provider method.

```dart
void _handleWidgetClick(Uri? uri) {
  if (uri == null) return;
  WidgetsBinding.instance.addPostFrameCallback((_) {
    if (!mounted) return;
    _dispatchWidgetUri(uri);
  });
}

void _dispatchWidgetUri(Uri uri) {
  switch (uri.host) {
    case 'record':
      _switchToChatTab();
      _focusChatInput();
      break;
    case 'camera':
      _switchToChatTab();
      context.read<ChatProvider>().pickImageFromCamera(context: context);
      break;
  }
}
```

### Step 3: Add the 200 ms defensive retry (only if needed)
- If during manual testing the cold-start `camera` path occasionally lands on ChatTab but no picker opens, add a single retry:

```dart
if (!_chatProviderReady()) {
  await Future.delayed(const Duration(milliseconds: 200));
  if (!mounted) return;
}
```

- `_chatProviderReady()` can be implemented as a try-catch around `context.read<ChatProvider>()` or a check on a known property.
- If the retry never fires during testing, remove it before merge — only keep what's verifiably needed.

### Step 4: Test the onboarding interplay
- If the user has not completed onboarding AND taps the widget: today, `addPostFrameCallback` shows onboarding before the widget callback runs (because both are in `initState`). Verify this order is correct — onboarding-first is the right UX (we don't want to open the camera before the user has been onboarded).
- If the widget callback fires *during* onboarding (race), defer it until onboarding completes by awaiting the dialog before dispatching the widget Uri.
