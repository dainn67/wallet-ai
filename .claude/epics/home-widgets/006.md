## Context

T002 rewrote the Glance composables; T005 wired up the cold-start flow. This task is the verification gate: manually QA the widget across 5 breakpoints × 3 Android API levels, stopwatch the NFR-1 latency target, and run an NFR-3 regression check on the data refresh pipeline. Findings are captured in `docs/features/home-widget.md` for future reference.

## Description

Three parallel verification streams:

1. **Layout & interaction matrix** (FR-1, FR-2, FR-3, FR-5): For each combination of {1×1, 1×2, 2×1, 2×2, 3×2+} × {API 26, 30, 34}, place the widget, screenshot it, and tap every interactive element. Record outcomes in a results matrix in the doc.
2. **Performance (NFR-1):** Stopwatch tap-to-ready latency for cold-start and warm-resume paths on a mid-range emulator profile (Pixel 4 / 4GB RAM). 5 trials per layout, record p50 and p95.
3. **State freshness regression (NFR-3):** Create, edit, and delete a record in-app. Verify widget data (balance, income, spent) updates within 30 s without manual refresh.

If any breakpoint fails NFR-2 (overflow/clipping) — most likely 2×2 with the new ActionIconRow + stats — escalate immediately. Two fallback paths documented in the epic: (a) move stats to 3×2+ only, or (b) shrink ActionIconRow icons to 36 dp.

## Acceptance Criteria

- [ ] **FR-1–5 / All interaction matrix cells pass:** 5 layouts × 3 APIs = 15 cells; every cell has all expected affordances visible and tappable.
- [ ] **NFR-1 / p95 tap-to-ready ≤ 800 ms (physical device):** On a mid-range device or equivalent emulator, 5 trials per (camera, write, bar) path → p95 must be within 800 ms.
- [ ] **NFR-2 / No overflow or clipping:** All 5 breakpoints render cleanly on API 26, 30, 34. Screenshots attached to results doc.
- [ ] **NFR-3 / State freshness preserved:** After in-app record CRUD, widget data updates within 30 s on physical device.
- [ ] **Results documented:** `docs/features/home-widget.md` has a "QA results" section with the matrix, screenshots (linked or embedded), latency table, and any escalations.

## Implementation Steps

### Step 1: Set up emulator profiles
- Create 3 Android emulator profiles: `pixel_4_api_26`, `pixel_4_api_30`, `pixel_4_api_34` (4 GB RAM each).
- Boot each, install the debug APK.

### Step 2: Layout & interaction matrix (15 cells)
- For each (layout × API) combination:
  - Place widget at the target size.
  - Take a screenshot (`adb shell screencap -p /sdcard/wally_<api>_<layout>.png && adb pull ...`).
  - Tap every affordance: bar, write icon (if present), camera icon (if present), root fallback (a non-icon area).
  - Record pass/fail in a markdown table in `docs/features/home-widget.md`:

```markdown
| Layout | API 26 | API 30 | API 34 |
|---|---|---|---|
| 1×1 SMALL | ✓ | ✓ | ✓ |
| 1×2 TALL | ✓ | ✓ | ✓ |
| ... | ... | ... | ... |
```

### Step 3: NFR-1 latency stopwatch
- Use `adb shell input tap <x> <y>` followed by a screen recording (`adb shell screenrecord ...`) — frame-count the recording to derive milliseconds (30 fps = 33 ms/frame).
- Run 5 trials each for:
  - Cold-start → camera (tap camera icon, measure tap to picker visible)
  - Cold-start → text input (tap bar, measure tap to keyboard visible)
  - Warm-resume → camera
  - Warm-resume → text input
- Record p50 and p95 in a small table:

```markdown
| Path | p50 (ms) | p95 (ms) | Target | Pass? |
|---|---|---|---|---|
| Cold → camera | 920 | 1340 | ≤800 (warm) / ≤1500 (cold) | ✓ |
| ... |
```

### Step 4: NFR-3 regression
- On a clean device:
  - Place 2×2 widget on home screen, note balance.
  - Open app, create a new expense record.
  - Press home, view widget within 30 s — balance/spent updated.
  - Repeat for edit and delete operations.
- Record pass/fail in the doc.

### Step 5: Escalations (if any)
- If overflow on 2×2 → trigger Risk Mitigation #5 from epic. Document the chosen fallback (stats to 3×2+ only, OR shrink ActionIconRow icons) and link to a follow-up task or epic addendum.
- If NFR-1 p95 fails on physical device → diagnose: app cold-start time? Provider init? Add findings to doc.
