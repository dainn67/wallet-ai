## Context

The widget's new camera icon (T002) fires a `homeWidget://camera` Uri intent. The Flutter side must catch that Uri and route to ChatTab + trigger the camera picker. Today `HomeScreen._handleWidgetClick` only handles `homeWidget://record`.

## Description

Extend `_handleWidgetClick(Uri? uri)` in `lib/screens/home/home_screen.dart` with a new branch for `homeWidget://camera`. The branch:

1. Switches to ChatTab.
2. Calls `context.read<ChatProvider>().pickImageFromCamera()` (method to be added by T004).

T003 can be merged before T004 — the method call will compile-fail temporarily, so leave a TODO comment with the planned signature so the file still compiles by stubbing the call inside `// TODO(T004):` if needed, OR coordinate the merge order with T004.

Also add a no-op branch for `homeWidget://open` so the root fallback URI from T002 lands gracefully.

## Acceptance Criteria

- [ ] **FR-4 / Camera URI routes to ChatTab + camera:** A widget intent with Uri `homeWidget://camera` lands in `_handleWidgetClick` → switches to ChatTab → invokes the camera picker.
- [ ] **FR-4 / Record URI still works (no regression):** Existing `homeWidget://record` flow is unchanged — ChatTab opens with text input focused.
- [ ] **FR-5 (indirect) / Open URI is a graceful no-op:** Receiving `homeWidget://open` brings the app to foreground and does nothing else (no exception, no tab switch from the user's last view).
