# Task: Render emoji in CategoryWidget, RecordWidget, SuggestionBanner + perf/visual QA + docs

## Context
With the emoji column populated (task 211), the user-facing payoff is rendering it everywhere a category name appears today: the Categories tab list, the record card subtitle, and the chat suggestion banner. This is the task that delivers the actual visual improvement the PRD is buying. NFR-2 (≤5ms first-frame regression on RecordsTab/CategoriesTab) and NFR-3 (cross-platform parity, no tofu boxes) are validated as part of this task because they only become measurable once rendering is live. The Living Docs mandate also requires updating `docs/features/` and `project_context/architecture.md`.

## Description
Prepend the emoji to the visual surface in three places:
- **`CategoryWidget`**: render `Text(category.emoji, style: TextStyle(fontSize: 20))` as the leading element inside the row, before the existing income/expense direction icon. The directional arrow stays — it conveys direction (in/out), not category identity, so the two signals are complementary.
- **`RecordWidget`**: prefix the category-name subtitle with `'${category.emoji} '` so the final string reads e.g. `🍔 Food & Drink • Cash`.
- **`SuggestionBanner`**: no client change needed. The server-generated `suggestion.message` (italic body of the banner) will carry the emoji inline (e.g., `"Should I create a new one called 🚌 'Commute'?"`). Document this contract explicitly in the file with a one-line comment so future readers don't re-add a client-side emoji injection.

Run a Flutter dev profiler trace on RecordsTab and CategoriesTab before/after the changes to verify NFR-2 (Δ ≤ 5ms first-frame). Manually QA on both iOS (Apple Color Emoji) and Android (Noto Color Emoji) to verify NFR-3 — confirm no tofu/missing-glyph for any AD-5 emoji or for 🏷️. Update `docs/features/category-icons.md` (new) and `project_context/architecture.md` per the Living Docs mandate.

## Acceptance Criteria

- [ ] **FR-5 / Categories tab:** Each row in CategoriesTab shows `category.emoji` as a leading `Text` at font size 20, before the existing direction-icon container.
- [ ] **FR-5 / Records list:** Every record card's category subtitle is prefixed with `'${category.emoji} '` — e.g. `🍔 Food & Drink • Cash`.
- [ ] **FR-5 / Suggestion banner:** The italic suggestion message rendered in `SuggestionBanner` includes the proposed emoji inline (delivered via server's `message` string). A code comment in `suggestion_banner.dart` documents this server-side contract.
- [ ] **FR-5 / Fallback renders:** A category whose `emoji = '🏷️'` (e.g., user-created with no emoji) renders the 🏷️ glyph at the appropriate site, never an empty space.
- [ ] **NFR-2 / Performance budget:** First-frame render time on RecordsTab and CategoriesTab regresses by ≤5ms (averaged over 10 cold opens, measured via Flutter dev profiler). Trace results recorded in `tests/integration/epic_category_icons/perf-notes.md`.
- [ ] **NFR-3 / iOS parity:** All seed emojis (AD-5 map) and 🏷️ render as native Apple Color Emoji on an iOS device — no tofu, no question-mark glyph.
- [ ] **NFR-3 / Android parity:** Same as above on an Android device with Noto Color Emoji (or vendor variant).
- [ ] **Living Docs:** `docs/features/category-icons.md` exists describing user-visible behavior. `project_context/architecture.md` mentions the Category.emoji field, the v9→v10 migration, and the three render sites.

## Interface Contract

## Technical Details

- **Approach:** Mirrors AD-1 (emoji is just a `String` on the model, rendered as a plain `Text`). No `Image`, no `SvgPicture`, no asset bundle. The platform's color emoji font does the heavy lifting.
- **Patterns to follow:**
  - `lib/components/category_widget.dart:54-58` — the existing icon container shows the row layout pattern.
  - `lib/components/record_widget.dart` — find the existing subtitle composition; mirror its style.
- **Why suggestion banner needs no client change:** the banner already renders `widget.suggestion.message` verbatim (italic style), and `message` is fully server-controlled. The server's prompt + validator (task 212) produces a message string that includes the chosen emoji inline. This keeps the client UI logic simple and gives the server full control over phrasing.
- **Edge cases:**
  - Emoji renders larger than expected on some Android skins — keep the explicit `fontSize: 20` to stabilize.
  - Right-to-left locales — `Text(category.emoji)` respects `TextDirection.rtl`; the row order should still be emoji → name. If issues surface in RTL QA, wrap the leading emoji in `Directionality(textDirection: TextDirection.ltr, ...)`.
  - RecordWidget subtitle truncation on narrow screens — the existing `TextOverflow.ellipsis` handles this; emoji adds ~1.5 chars worth of width, negligible.
