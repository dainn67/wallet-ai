# Task 001: Dependencies and platform permission manifests

## Context

Wally AI currently has no image-picking or image-compression capability. This task installs the two native Flutter plugins (`image_picker`, `flutter_image_compress`) and registers the OS-level permission strings so that all downstream tasks can rely on them. Nothing else in the image-input epic can be built or tested without this foundation in place.

## Description

Add `image_picker` and `flutter_image_compress` to `pubspec.yaml`. Write the required permission description strings into iOS `Info.plist` and the camera permission into `AndroidManifest.xml`. Verify the project still builds on both platforms after these additions.

`image_picker` handles gallery and camera picking with built-in lazy permission prompts on first use (no separate `permission_handler` needed — see Architecture Decision AD-3 in the epic). `flutter_image_compress` provides native HEIC decoding + JPEG re-encoding needed by T010.

## Acceptance Criteria

- [ ] **NFR-3 / iOS parity:** `image_picker` and `flutter_image_compress` resolve in `fvm flutter pub get` with no SDK constraint conflicts against `^3.9.2`.
- [ ] **NFR-3 / iOS manifest:** `ios/Runner/Info.plist` contains both `NSCameraUsageDescription` and `NSPhotoLibraryUsageDescription` with non-empty, user-facing strings in English (Vietnamese copy optional at this stage).
- [ ] **NFR-3 / Android manifest:** `android/app/src/main/AndroidManifest.xml` contains `<uses-permission android:name="android.permission.CAMERA" />`.
- [ ] **NFR-3 / Build:** `fvm flutter build apk --debug` and `fvm flutter build ios --debug --no-codesign` both complete without error.
- [ ] **NFR-4 / No prompt at launch:** Cold-launching the app on a device or simulator shows no permission dialog before any user action in the ChatTab. (Verify manually after the iOS build passes.)

## Implementation Steps

### Step 1: Add dependencies to `pubspec.yaml`

- Open `pubspec.yaml`.
- Under the `dependencies:` block (alongside existing `provider`, `sqflite`, etc.), add:
  ```yaml
  image_picker: ^1.1.2
  flutter_image_compress: ^2.3.0
  ```
- Check pub.dev before pinning if either package has a newer stable release compatible with Dart SDK `^3.9.2`.
- Run `fvm flutter pub get` and confirm no version conflicts.

### Step 2: Add iOS permission strings to `ios/Runner/Info.plist`

- Open `ios/Runner/Info.plist`.
- Add the following two key-value pairs inside the root `<dict>`:
  ```xml
  <key>NSCameraUsageDescription</key>
  <string>Wally needs camera access to capture receipts and bills for logging transactions.</string>
  <key>NSPhotoLibraryUsageDescription</key>
  <string>Wally needs photo library access to attach receipts and bank notification screenshots.</string>
  ```
- Place them alongside existing permission keys (search for `NSPhotoLibrary` first to avoid duplicates).

### Step 3: Add camera permission to `android/app/src/main/AndroidManifest.xml`

- Open `android/app/src/main/AndroidManifest.xml`.
- Add before the `<application>` tag:
  ```xml
  <uses-permission android:name="android.permission.CAMERA" />
  ```
- Note: `READ_MEDIA_IMAGES` (Android 13+) and `READ_EXTERNAL_STORAGE` (Android < 13) are handled automatically by `image_picker`'s AAR — no manual entry needed.

### Step 4: Verify builds

- Run `fvm flutter build apk --debug` — must succeed.
- Run `fvm flutter build ios --debug --no-codesign` — must succeed.
- If build errors surface due to `flutter_image_compress` native compilation (common on older Xcode), check the package's README for any additional iOS `Podfile` setup (e.g., minimum iOS version bump to 12.0 if needed).

## Technical Details

- **Approach:** Pure configuration — no Dart code changes in this task.
- **Plugin versions:** Pin to exact minor versions to avoid breaking pub upgrades. Document in commit message which versions were selected and why.
- **flutter_image_compress iOS note:** The package requires iOS 12.0+. Check `ios/Podfile` for `platform :ios, '12.0'` or higher; bump if needed.
- **No `permission_handler` dependency:** Per AD-3, `image_picker` supplies its own permission prompt flow. Adding `permission_handler` would be redundant and adds surface area. Do not add it.
