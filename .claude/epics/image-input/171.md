# Task 010: ImageProcessingService — compression, pass-through, and base64 encode

## Context

Before any image can be uploaded to the `/streaming` endpoint, it must be decoded, auto-rotated (EXIF), resized to ≤1600px on the longest edge, re-encoded as JPEG quality 85, and base64-encoded. HEIC inputs (common on iOS) must be converted to JPEG. Images that are already small (longest edge ≤1600px AND size ≤500 KB) skip re-encoding to save CPU. Any image that remains >1.5 MB after compression is rejected with an `OversizeImageException`. This service is pure computation with no state — tested easily via mock fixtures.

## Description

Create `lib/services/image_processing_service.dart` as a singleton matching the project's existing `setMockInstance` pattern (see `ChatApiService` for reference). Implement `processPickedImage(XFile)` and `toBase64(Uint8List)`. Write unit tests with fixture images covering the four key scenarios: large JPEG (compress), small JPEG (pass-through), HEIC (decode+re-encode), and pathological-oversize (reject).

Architecture Decision AD-5 (pass-through rule) and AD-1 (compress on pick) from the epic define the behavior implemented here.

## Acceptance Criteria

- [ ] **FR-4 / Happy path — large photo compressed:** Given a JPEG with longest edge >1600px, `processPickedImage` returns bytes with longest edge ≤1600px and size ≤1.5 MB.
- [ ] **FR-4 / HEIC input:** Given an HEIC `XFile`, returned bytes are valid JPEG (magic bytes `FF D8 FF`) — not HEIC.
- [ ] **FR-4 / Small image pass-through:** Given a JPEG with longest edge ≤1600px AND size ≤500 KB, the service returns bytes without calling `flutter_image_compress` (verify by checking returned size is within 5% of input).
- [ ] **FR-4 / Edge — oversize after compression:** Given a fixture that stays >1.5 MB post-compression, `processPickedImage` throws `OversizeImageException`.
- [ ] **NFR-1 / Upload size budget:** Processing 5 images with the large-JPEG fixture in parallel via `Future.wait` produces a total payload ≤8 MB unencoded (measured in test).
- [ ] **NFR-2 / Compression perf:** Compressing a single 4000×3000 JPEG completes in ≤2.5s on CI (unit test with `Stopwatch`; flag as warning if exceeded — not hard failure on CI).

## Implementation Steps

### Step 1: Define the service and data types

- Create `lib/services/image_processing_service.dart`.
- Define `OversizeImageException` as a plain class extending `Exception`:
  ```dart
  class OversizeImageException implements Exception {
    final int sizeBytes;
    OversizeImageException(this.sizeBytes);
    @override
    String toString() => 'Image too large after compression: ${sizeBytes ~/ 1024}KB (max 1536KB)';
  }
  ```
- Define `ProcessedImage`:
  ```dart
  class ProcessedImage {
    final Uint8List bytes;
    final int sizeBytes;
    ProcessedImage({required this.bytes}) : sizeBytes = bytes.length;
  }
  ```
- Implement the singleton pattern (copy from `ChatApiService`):
  ```dart
  class ImageProcessingService {
    static final _instance = ImageProcessingService._internal();
    static ImageProcessingService? _mockInstance;
    factory ImageProcessingService() => _mockInstance ?? _instance;
    ImageProcessingService._internal();
    static void setMockInstance(ImageProcessingService? m) => _mockInstance = m;
  ```

### Step 2: Implement `processPickedImage(XFile file)`

- Read bytes from `file.readAsBytes()`.
- Apply pass-through rule (AD-5): decode image dimensions using `flutter_image_compress`'s `testCompressAndGetFile` or, more cheaply, read only the header. **Simpler alternative:** read byte length first — if `bytes.length <= 500 * 1024`, tentatively skip compression; after decoding confirm longest edge ≤1600. If either condition fails, proceed to compression.
  - **Recommended pragmatic approach:** always compress HEIC (no pass-through); for JPEG/PNG, pass through only if `bytes.length <= 512 * 1024` (close enough to 500KB, avoids decoding just to read dimensions).
- Compress using `FlutterImageCompress.compressWithList`:
  ```dart
  final compressed = await FlutterImageCompress.compressWithList(
    bytes,
    minWidth: 1600,
    minHeight: 1600,
    quality: 85,
    format: CompressFormat.jpeg,
    keepExif: false,  // strip EXIF to remove location data and apply rotation
  );
  ```
  - Note: `minWidth`/`minHeight` in `flutter_image_compress` cap the longest edge when both are set equal. Verify this behavior in the plugin docs; adjust if the API differs.
- If `compressed.length > 1.5 * 1024 * 1024` → throw `OversizeImageException(compressed.length)`.
- Return `ProcessedImage(bytes: Uint8List.fromList(compressed))`.

### Step 3: Implement `toBase64(Uint8List bytes)`

- Single-line: `return base64Encode(bytes);`
- Exposed as a method (not a top-level function) so it can be mocked in tests of `ChatProvider`.

### Step 4: Export from services barrel

- Open `lib/services/services.dart`.
- Add: `export 'image_processing_service.dart';`

- Create `test/services/image_processing_service_test.dart`.
- Create `test/fixtures/images/` and add:

  - `large.jpg` — any JPEG ≥2 MB, longest edge >2000px
  - `receipt.heic` — a real HEIC file, or generate one in the test using `flutter_image_compress`'s test utilities


## Technical Details

- **`flutter_image_compress` API:** Use `compressWithList(Uint8List, {minWidth, minHeight, quality, format})`. Setting both `minWidth` and `minHeight` to 1600 caps the longest edge. Verify with a 4000×3000 image that output longest edge ≤1600.
- **EXIF stripping:** Set `keepExif: false` to strip location data and apply rotation correction. This is important for privacy (receipts may contain location EXIF) and for correct orientation in the bubble (AD-1: compress on pick so rotation is applied before display).
- **HEIC handling:** `flutter_image_compress` handles HEIC natively on both iOS and Android when `format: CompressFormat.jpeg` is set. No special branch needed.
- **Async safety:** `compressWithList` is async and runs natively — no need to wrap in `compute()` for this plugin. If UI jank is observed during T020 testing, revisit.
- **Edge cases:** Corrupt/unreadable files from the picker will throw from `readAsBytes()` — let the exception propagate; callers in T005 (chat_tab) catch and display inline error.
