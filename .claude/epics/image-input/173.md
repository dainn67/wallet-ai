# Task 012: Model, provider, and API payload wiring

## Context

Wally's chat currently only sends text to the `/streaming` endpoint. This task extends the data model, provider, and network service to carry images through the whole pipeline — from the user tapping send, to the outbound JSON body, to the existing stream parser. The response parsing (`ChatProvider._handleStream`) is **not changed** — the server returns the same record-array JSON as for text, so the existing `onDone` handler works without modification. This is the central cross-cutting change; all UI tasks (T020, T021) depend on it.

Architecture Decisions from the epic that apply here:
- **AD-2:** `images` is a top-level JSON field (sibling of `query`), array of base64 JPEG strings. Omitted entirely when empty.
- **AD-4:** `ChatMessage` carries a transient `List<Uint8List>? imageBytes` field — not serialized to JSON (parallels `Record.suggestedCategory`).
- **AD-6:** If caption is non-empty but all images failed compression, send as text-only. If both are empty, no-op.

## Description

Three coordinated changes:

1. **`ChatMessage`** — add nullable `List<Uint8List>? imageBytes`, update constructor and `copyWith`. Keep `toJson`/`fromJson` unchanged (transient field, never persisted).
2. **`ChatApiService.streamChat`** — accept optional `List<String>? imagesBase64`; when non-empty, add top-level `'images': imagesBase64` to the POST body.
3. **`ChatProvider.sendMessage`** — accept optional `List<Uint8List>? imageBytes`; base64-encode them via `ImageProcessingService.toBase64` before passing to `streamChat`.

Update tests for the new signatures.

## Acceptance Criteria

- [ ] **FR-5 / Happy path — images + caption:** `ChatProvider.sendMessage('lunch', imageBytes: [bytes])` causes `ChatApiService.streamChat` to be called with `imagesBase64: ['<base64>']`, and the outbound POST body contains top-level `images: ['<base64>']` alongside `query: 'lunch'`.
- [ ] **FR-5 / Images-only (empty caption):** `sendMessage('', imageBytes: [bytes])` still fires the stream with the `images` field present; `query` is sent as empty string (not omitted, matching existing behavior where `message` is always passed).
- [ ] **FR-5 / Text-only backward compat:** `sendMessage('hello')` with no `imageBytes` produces a POST body with no `images` key — identical to current behavior (no regression).
- [ ] **FR-5 / Response parsing unchanged:** Streaming response parsed by existing `_handleStream` `onDone` — records appear in `ChatProvider.messages` without any change to parsing code (verified by existing tests still passing).
- [ ] **FR-7 / Error bubbles unchanged:** When stream `onError` fires (simulated in test), the assistant message content is updated with the error suffix — same as today (no regression).
- [ ] **AD-4 / Transient field:** `ChatMessage.toJson()` on a message with `imageBytes` does NOT include an `imageBytes` key (verified in unit test).
- [ ] **AD-6 / Both empty — no-op:** `sendMessage('', imageBytes: [])` returns immediately without adding a message or calling the API.

## Technical Details

- **Why top-level `images` (AD-2):** The current payload structure is `{ 'user': '123', 'query': msg, 'inputs': { ... } }`. The `inputs` dict is a Dify "variables" map used in prompt templating. Binary-like content (base64 images) does not belong in variables. Top-level `images` is the correct Dify convention for file inputs. Server team confirmed field name `images`.
- **Transient field pattern (AD-4):** `Record.suggestedCategory` in `lib/models/record.dart` is the existing precedent. The field is set on construction and used by the UI layer but never goes to SQLite. Same approach here — `ChatMessage.imageBytes` is ephemeral; it lives in memory for the duration of the chat session.
- **Empty-string `query` (resolves PRD W5):** The current `streamChat` always passes `'query': message` even if message is empty (the method simply doesn't guard). Maintaining this behavior is intentional — server is already tolerant of empty `query` when `images` is present.
- **Existing callers of `sendMessage`:** Only `lib/screens/home/tabs/chat_tab.dart` calls it. It currently passes positional text only — `sendMessage(text)` remains valid after adding the optional `imageBytes` named param. Update T020 to pass `imageBytes` alongside.
