## Recommendations

1. **Define the full seed emoji map upfront.** The first implementation task will need it. A simple table (category_id | name | emoji) in FR-1 removes an ambiguity that will otherwise surface as a code review comment on the first PR.

2. **Clarify the emoji input UX path explicitly.** The intent (plain TextField + OS keyboard first, picker optional) is clear to the author but buried in the Risk entry. Move that decision to FR-4 as a first-class implementation note. The Out of Scope entry already sets the right expectation — FR-4 just needs to match it.

3. **Consider a third user story for the new-user persona.** US-3 ("see an emoji on every existing category after upgrade") is a migration story and doesn't fully cover the new-user experience of seeing seeded emojis on first install. A short US-5 would close the persona gap and make the seed-map requirement's purpose more explicit.

4. **`updated: null` in frontmatter.** A cosmetic fix — set `updated` to the same timestamp as `created` on initial draft so tooling that reads `updated` doesn't have to handle null.
