# Product Brief: ai-pattern-analyze

## Vấn đề (Đã Reframe)
App có đầy đủ dữ liệu chi tiêu trong SQLite nhưng dữ liệu đó ở dạng raw rows — server AI không thể hiểu trực tiếp.
Cần một lớp transformation biến raw records thành một structured snapshot có ngữ nghĩa rõ ràng: bao nhiêu tiền chi vào category nào, vào thời điểm nào trong ngày, xu hướng 30 ngày ra sao.
Đây là data contract giữa Flutter app và Python AI server.

## Yêu cầu Gốc vs Hướng đi Mới
- **Gốc:** Build toàn bộ pipeline — sync strategies, SharedPreferences, server-side merging, background execution
- **Reframe:** Chỉ build `DataAggregatorService.buildSnapshot()` — một method thuần Dart trả về `Map<String, dynamic>` sạch, testable, server-ready
- **Insight chính:** RecordRepository đã có `getCategoryTotals(start, end)` và `getAllRecords()` với JOIN sẵn — aggregator chỉ cần transform, không query mới

## User Mục tiêu
**Chính:** Developer (bạn) — cần establish data contract trước khi viết server endpoint. Snapshot format là nguồn sự thật cho cả Flutter side lẫn Python side.

## Kết quả Mong muốn
Sau khi gọi `DataAggregatorService().buildSnapshot()`, nhận được một Map có thể `jsonEncode` trực tiếp rồi POST lên server.
Server nhận vào biết ngay: user này hay chi tiền vào category nào, buổi nào trong ngày, tháng này tổng bao nhiêu.
Không cần thêm xử lý hay mapping phía server.

## Quyết định Scope
**Mode:** REDUCE

- ✅ TRONG:
  - Class `DataAggregatorService` trong `lib/services/`
  - Method `buildSnapshot({bool isInitial = false})` → `Map<String, dynamic>`
  - Initial snapshot: toàn bộ history, group theo category + time-of-day
  - Daily snapshot: records 24h qua + summary stats 30 ngày
  - Reuse `RecordRepository.getCategoryTotals()` và `getAllRecords()`
  - Time-of-day buckets: Morning (5-11h), Afternoon (11-17h), Evening (17-22h), Night (22-5h)
  - Client metadata: current_time, timezone, language, sync_type

- ❌ NGOÀI:
  - Gọi API lên server (scope riêng: server-pattern-api)
  - Lưu/đọc SharedPreferences pattern (scope riêng: ai-sync-scheduler)
  - Trigger logic "24h một lần" (scope riêng: ai-sync-scheduler)
  - Background execution / WorkManager (optimize sau)
  - Server-side pattern merging (Python scope)
  - UI greeting / action chips (scope riêng: ai-home-suggestions)

## Bối cảnh Sản phẩm
- `RecordRepository` đã có: `getAllRecords()` (JOIN Category+MoneySource), `getCategoryTotals(start, end)`
- `StorageService` đã có pattern để lưu SharedPreferences — dùng cho follow-up epic
- `ApiService` và `ChatApiService` đã có pattern gọi server — dùng để build endpoint sau
- Epic `record-provider` (backlog) cung cấp data layer — `DataAggregatorService` phụ thuộc `RecordRepository` đã stable

## Đặt cược & Rủi ro Chính
**Đặt cược:** Server AI phân tích đúng pattern nếu có category + amount + time-of-day + 30-day summary
**Nếu sai:** Server cần thêm field (ví dụ: day-of-week, description text) → chỉ cần update `buildSnapshot()`, không ảnh hưởng trigger/sync logic
**Giảm thiểu:** Thiết kế snapshot map extensible — server ignore field thừa, app thêm field mới không breaking

## Quy mô PRD Đề xuất
**small** — 1 file service mới (~100-150 lines), 0 UI thay đổi, reuse repository hiện có hoàn toàn

## Câu hỏi Mở cho PRD Discovery
1. Initial sync giới hạn bao nhiêu ngày lịch sử? (Toàn bộ vs 90 ngày gần nhất)
2. Description field có đưa vào snapshot không? (Tăng AI insight nhưng tăng payload size)
3. Snapshot có cần group theo sub-category riêng hay chỉ parent category?
