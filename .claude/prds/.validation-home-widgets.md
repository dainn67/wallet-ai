## Warnings (should fix)

- [ ] **Correctness — US-1 AC contradicts FR-1:** US-1 acceptance criteria still says "all other layouts [non-1×1] show the label text alongside the icon." FR-1 was updated during prd-edit to extend the label-omission rule to any breakpoint ≤ 80 dp wide (including 1×2 TALL), but US-1's AC was not updated to match. A developer following US-1's AC would add a label to the 1×2 TALL layout, violating FR-1. Fix: update US-1 AC bullet to read "layouts wider than 80 dp show the label text alongside the icon; layouts ≤ 80 dp wide show the icon only."
  — Section: US-1 Acceptance Criteria

- [ ] **Coherence — FR-4 missing cold-start scenario:** The only scenario for FR-4 (deep-link routing) covers "app already in background." It does not specify behaviour when the app is not running (cold start from widget tap). This is actually the more common case. Without a scenario, an implementer might handle background-resume correctly but break cold-start routing (e.g. the intent extras are lost during Flutter engine init). Fix: add a cold-start scenario — GIVEN the app is not running, WHEN `homeWidget://camera` intent arrives via widget tap, THEN the app launches, initialises, and opens the camera picker without requiring an additional tap.
  — Section: FR-4

- [ ] **Coherence — "immediately" is ambiguous in FR-3 scenario:** The camera icon tap scenario says "the app opens and immediately presents the image picker / camera." "Immediately" has no numeric definition in this scenario. NFR-1's 800 ms threshold covers it implicitly, but a reviewer running the scenario as a test has no pass/fail threshold to apply. Fix: replace "immediately" with "within 800 ms end-to-end from tap (per NFR-1)."
  — Section: FR-3 Scenario
