## Warnings (should fix)

- [ ] **Completeness:** NTH-1 and NTH-2 have no GIVEN/WHEN/THEN scenarios — the quality standard requires ≥1 scenario for every FR *and* NTH. — Section: Requirements (NICE-TO-HAVE) — Fix: Add a brief scenario for each, even if it's the happy path only.

- [ ] **Completeness:** "Hands-occupied user" and "Multi-item logger" are defined personas but never appear in any User Story (US-1 through US-4 reference only "on-the-go recorder" or the generic "any user"). — Section: User Stories — Fix: Map US-2 or US-3 to "hands-occupied user"; add a story for "multi-item logger" or merge it into an existing one.

- [ ] **Completeness:** US-2, US-3, US-4 say "As any user" — "any user" is not a defined persona. — Section: User Stories — Fix: Replace with the closest matching persona (e.g. "As a WalletAI user" or pick one of the three defined personas).

- [ ] **Correctness:** NFR-1 contains contradictory values: "≤ 1 Mbps bitrate (e.g., AAC 128 kbps)". AAC 128 kbps = 0.128 Mbps, not 1 Mbps — nearly 8× apart. An implementer cannot know which limit to enforce. — Section: Non-Functional Requirements — Fix: Pick one value. Suggested: "AAC at 128 kbps; total encoded audio must not exceed 5 MB for a 30-second recording."

- [ ] **Correctness:** FR-4 "Voice + text" scenario is missing the WHEN clause — goes directly from GIVEN to THEN. — Section: FR-4 — Fix: Add "WHEN the user taps send" or equivalent.

- [ ] **Coherence:** The new request field carrying audio is never named in the PRD — referred to only as "a new field". The server requirements doc uses `audio` but the PRD itself leaves implementers to infer the name. — Section: FR-4 — Fix: Explicitly state the field name, e.g. `audio: string | null`.

- [ ] **Coherence:** FR-5 voice failure detection is vague — "if the AI returns no parseable records and indicates it couldn't understand" has no clear programmatic detection path. The client parser works on the `--//--` delimiter; an empty records array `[]` is the only signal available. A non-financial response may still return `[]` for normal chat messages. — Section: FR-5 — Fix: Clarify detection: e.g. "if the `audio` field was sent and the parsed records array is empty, show the voice error message."

- [ ] **Coherence:** Dependencies are missing owner and status fields. — Section: Dependencies — Fix: Add owner (e.g. "client team" / "server team" / "TBD") and status (pending/resolved/blocked) for each dependency per the standard format.

- [ ] **Coherence:** US-4 (image error feedback) belongs to the already-merged `image-input` epic, not `voice-input`. Including it here creates scope confusion and may result in duplicate or conflicting implementations. — Section: User Stories / FR-5 — Fix: Either move image error handling to a standalone bug-fix task, or add a note that FR-5's image clause is a retroactive fix to the image-input feature and call it out explicitly.

## Recommendations

1. **Microphone permission flow** — FR-1 mentions the mic icon is disabled if permission is denied, but there's no requirement for the permission request flow itself (rationale dialog, first-tap prompt, settings deep-link if permanently denied). Add at least one scenario covering the permission denied path for FR-1 or as a new FR.

2. **Recording state in UI while AI is processing** — After the user stops recording and the audio is sent, there's no specified UI state for "audio sent, waiting for AI response." Users may tap the mic again thinking nothing happened. Consider specifying a loading/pending state between send and first SSE chunk.

3. **Empty recording guard** — What happens if the user taps start and immediately taps stop (< 1 second of audio)? The PRD doesn't address minimum recording duration. Add a constraint or scenario (e.g. minimum 1 second, or show "Recording too short").
