## Warnings (should fix)

- [ ] **Coherence — Italic weight missing from Dependencies & Assumptions:**
  FR-2 requires 5 font files: Regular 400, **Regular Italic 400i**, Medium 500, SemiBold 600, Bold 700. The Dependencies section (line 328) still lists only `"TTF files 400/500/600/700"` — the 400i file is absent. The Assumptions section (line 308) has the same gap. An implementer relying on the dependency list to assemble the font download would miss the italic file and only discover it when FR-5's inline suggestion text renders with synthesized italics.
  — Section: Dependencies, Constraints & Assumptions
  — Fix: Change `400/500/600/700` → `400/400i/500/600/700` in both the Dependencies entry and the Assumptions paragraph.

- [ ] **Coherence — "per-source accent color" undefined in FR-6 Zone 3:**
  FR-6 Zone 3 says "horizontally scrolling row of money-source cards with per-source accent color." FR-1 defines a six-step category-accent ramp for categories, but no assignment rule for money sources is specified. An implementer must know: are money-source card accents drawn from the same six-step ramp (assigned sequentially or by source ID), hardcoded per source, or always the primary tint? This is not covered in FR-1, FR-6, or the Assumptions.
  — Section: FR-6, FR-1
  — Fix: Add one sentence to FR-6 Zone 3 specifying the color-assignment rule (e.g., "each money-source card uses a color from the six-step category-accent ramp assigned by index modulo 6" or "money-source cards default to `colorScheme.primaryContainer` tint in v1; per-source customization is NTH").

- [ ] **Coherence — "entity-semantic tint" undefined for transfer records in FR-6:**
  FR-6 transaction rows say "leading rounded icon-square with entity-semantic tint" and then maps income = green token, expense = red token. Transfer records exist in the system (`type: 'transfer'`) and currently render with an indigo `swap_horiz` glyph. The redesign spec doesn't define the icon-square tint for transfer rows — an implementer would have to guess. Given that the tertiary color `#EC4899` (pink) is taken by another role, and the current UI uses indigo, clarification is needed.
  — Section: FR-6
  — Fix: Add a third mapping line: "transfer = primary tint (or `colorScheme.tertiaryContainer` — implementer's choice from ramp); `swap_horiz` icon."

## Recommendations

1. **FR-8 per-popup design notes (carry-over):** Listing 11 popup surfaces under one shared "new design language" description still leaves task granularity to prd-parse's discretion. The epic decompose step can split these, but adding a 1-line note per popup (e.g., "TransferPopup — replace flat action row with pill primary button + outlined secondary") would produce better-scoped tasks.

2. **SC-1 overlap with SC-5:** Success Criteria 1 ("zero surfaces using pre-redesign defaults — maker sign-off") and SC-5 ("5-point checklist — maker review of screenshots") both involve a maker sign-off visual review. They are complementary but redundant if the SC-5 checklist is the mechanism for SC-1. Consider folding SC-1's sign-off condition into SC-5's checklist as point (f).

3. **NFR-5 surface list is narrow:** NFR-5's WCAG contrast threshold lists only 5 specific surfaces (input bar, nav tabs, edit icon, popup actions, expand chevron). The app bar wordmark `#8B5CF6` on white `#FFFFFF` should be explicitly included — `#8B5CF6` on white passes AA at ~4.6:1, but only barely. Worth calling out so it's verified and not accidentally adjusted.
