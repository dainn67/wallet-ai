## Warnings (should fix)

- [ ] **Coherence — No-action prompt leaves remaining prompts inaccessible:** FR-3 (no-action scenario) correctly specifies that when a prompt with `actions: []` is tapped, "chip bar disappears entirely." However, this means any remaining unconsumed prompts become completely unreachable until the user sends the current message. This UX consequence is not acknowledged anywhere in the PRD, and `prd-parse` may not account for it. Example: if prompts = ["Bánh mì" (no actions), "Cà phê"], user taps "Bánh mì" → chip bar gone → "Cà phê" is now inaccessible unless user sends first. This may be intentional but is silent in the spec.
  - **Section:** FR-3, Scenario: Prompt with no actions
  - **Fix:** Add a note to FR-3: *"When a no-action prompt is selected, remaining prompts become temporarily inaccessible until the user sends. This is intentional — the user is committed to logging the selected expense."* Or, alternatively, spec a different behavior: when a no-action prompt is tapped, show remaining prompts (minus the selected one) after a short delay, allowing switching before send.

- [ ] **Coherence — Occasional Logger persona has no dedicated story:** The "Occasional Logger" persona has a distinct primary need ("be reminded of their recurring expenses") that is semantically different from Daily Logger ("log under 5 seconds"). All 4 User Stories address Daily Logger's speed goal. No story captures the Occasional Logger's discovery/reminder value — e.g., "As an occasional returning user, I want to see my usual spending habits suggested so that I don't have to recall them from memory."
  - **Section:** User Stories
  - **Fix:** Either (a) add a US-5 for Occasional Logger's reminder use case, or (b) update US-1 acceptance criteria to explicitly include "prompts serve as a reminder of previous spending habits, not just a speed shortcut."

## Recommendations

1. **FR-5 empty-send:** The input-clear scenario implies sending an empty message removes the active prompt. If the send button is disabled when input is empty (likely existing behavior), this scenario is untestable as written. Confirm whether empty-send is blocked upstream and note it.
2. **NFR-1 "standard device":** Still vague. Adding "(e.g., mid-range Android, ~4GB RAM)" would make QA reproducible.
3. **Logging speed criterion:** The ≤5s target is measured "from greeting load" — this implies the greeting must already be completed before the timer starts. Make explicit whether the 5s starts from (a) app open, (b) greeting stream complete, or (c) chip bar rendered.
