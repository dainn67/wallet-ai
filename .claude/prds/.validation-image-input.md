## Warnings (should fix)

- [ ] **Correctness:** Success criteria "Upload success rate" and "Compression effectiveness" are measured via "client-logged counters/sizes" — but no analytics/telemetry requirement is specified anywhere in the PRD. The app stack (SharedPreferences + SQLite) has no instrumentation layer. These criteria cannot be captured in production as written. — Section: Success Criteria — Fix: Either add an NFR for basic client telemetry, or change measurement to "validated in QA/test-bench" (acceptable for a v1 feature).

- [ ] **Correctness:** "Adoption: ≥25% of active chat users" is measured via server-side counts, but server-side analytics infrastructure is not listed as a dependency. If the server team has no logging on `/streaming` requests per-field, this metric is unmeasurable. — Section: Dependencies + Success Criteria — Fix: Add a dependency entry "Server-side analytics / request logging for `images` field detection — server team — Status: pending" so ownership is explicit.

- [ ] **Coherence:** FR-4 small-image pass-through behavior uses an ambiguous OR: "The image is not re-encoded (or is re-encoded losslessly enough to produce a similar size)" — these are two different code paths and produce different test outcomes. — Section: FR-4 / Scenario "Small image — pass through" — Fix: Pick one definitive behavior. Recommended: "Images whose longest edge is ≤1600px AND whose size is ≤500 KB skip re-encoding; all others are compressed."

- [ ] **Coherence:** Missing edge case — when ALL attached images fail the 1.5 MB post-compression cap (FR-4), it's unspecified whether the send is blocked entirely or the message sends with caption text only. A developer will have to guess. — Section: FR-4 — Fix: Add a scenario: "GIVEN all 3 attached images exceed 1.5 MB after compression WHEN the user taps send THEN send is blocked with an inline error AND no request is made."

- [ ] **Coherence:** FR-5, "images-only message" scenario: "`query` field is sent as empty string (or omitted per existing text-flow convention)" — the ambiguity (empty string vs. omitted) will force the developer to inspect `ChatApiService` themselves, and the server may treat these differently. — Section: FR-5 — Fix: Explicitly check `ChatApiService.sendStreamingMessage`'s current convention for empty text and specify one behavior. E.g., "query field is omitted when caption is empty string."

- [ ] **Coherence:** US-2 Acceptance Criteria: "a record with correct amount and a reasonable category guess" — "reasonable" is not testable in QA. A QA engineer cannot verify this without a rubric. — Section: User Stories / US-2 — Fix: Replace with "at least one record is created with a non-null `amount` field; category may be uncategorized (category_id: -1 is acceptable)."

- [ ] **Coherence:** NTH-2 ("Retry button on error bubble") vs Out of Scope ("Retry-queue for failed uploads") — a reader can misread these as contradictory (both mention retry). — Section: Out of Scope — Fix: Change Out of Scope entry to "No auto-retry queue (automatic re-send without user action) — a one-shot manual retry button is tracked as NTH-2."

## Recommendations

1. **FR-7 coverage gap:** The FR-7 scenarios cover "upload timeout" and "server error response" but not "server returns HTTP 200 with empty record array and no error payload" (a silent extraction failure). If the server can return a successful response with no records and no error text, the user would see an empty streaming response — confusing. Consider adding a scenario or at minimum a note in FR-7 about how this case is handled.

2. **No mention of loading state during upload:** The PRD says "same spinner / 'thinking' state as normal chat" in the blueprint but this is not captured as an FR or AC. It's a small UX gap that a developer might miss. Consider adding a sub-point to FR-5 or FR-6.

3. **Fullscreen image viewer scope:** FR-6 says "at minimum a simple fullscreen preview" — this is deliberately vague, but it means the developer could ship a bare `Image.network` in a scaffold and that's compliant. If the UX expectation is higher (e.g., pinch-to-zoom, swipe between images), that should be explicit. If basic is truly fine, the current wording is acceptable.

4. **`flutter_image_compress` Android HEIC support:** NFR-3 states "HEIC decoding works natively on both [platforms]." Android devices rarely produce HEIC natively (it's primarily Apple's format), but the claim is about *decoding* HEIC input on Android (e.g., if a user transfers HEIC files from iCloud). Confirm `flutter_image_compress` handles this on Android before claiming parity, or scope the HEIC guarantee to iOS only.
