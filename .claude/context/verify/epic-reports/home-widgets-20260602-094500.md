- **Category:** 4 — Missing Requirement

### Gap #4: Camera permission path diverged from PRD spec
- **Category:** 1 — Integration Gap
- **Severity:** **Medium**
- **Related issues:** #222 (T004)
- **Description:** T004's PRD/task spec called for explicit `Permission.camera.request()` via `permission_handler` with a context-gated in-app denial dialog (AC: "Permission flow: First-time permission request still shows the OS prompt; subsequent denials show the existing in-app dialog ONLY when context != null"). The agent followed the existing `ImagePickerService` codebase pattern instead, which delegates to `image_picker`'s native permission flow and never calls `permission_handler`. This was a deliberate, documented deviation — but it means the "subsequent denials show in-app dialog" AC is **not implemented**.
- **Evidence:** T004 handoff explicitly calls this out: "the actual codebase's `ImagePickerService` has a `// no extra permission logic needed per AD-3` comment ... I followed the existing codebase pattern (no manual `permission_handler` call)."
- **Recommendation:** Either accept as technical debt (the existing pattern is consistent, the OS still prompts on first denial) OR add a follow-up task to wire `permission_handler` and the in-app dialog. PRD spec was over-specified; codebase reality is acceptable.
- **Estimated effort:** Small (accept) / Medium (implement)

### Gap #5: Glance click priority unverified on a running widget
- **Category:** 3 — Phantom Completion (risk)
- **Severity:** **Medium**
- **Related issues:** #220 (T002), #224 (T006)
- **Description:** Epic Risk #2 ("Glance clickable overlap — root captures icon taps") is explicitly flagged as the second-highest risk. T002 placed inner clickables before root in source order based on a reading of how Glance composes modifiers, but this was **not verified by running the widget on an emulator and tapping the icons**. If the root captures icon taps, the write/camera icons become dead — FR-2 and FR-3 silently fail.
- **Evidence:** T002 handoff: "Inner clickables (bar + icons) are declared inside child composables before the root modifier is applied, so Glance's leaf-to-root hit-test gives them priority." This is reasoning, not observation.
- **Recommendation:** Make this the first verification in T006: place a 2×2 widget, tap each icon, observe Logcat for the correct URI per tap. If wrong, restructure modifier application.
- **Estimated effort:** Small (verify) / Medium (fix if broken)
